## 1. 问题本质定义

### 1.1 传统数据丢失 vs AI 善意破坏

```
传统数据丢失                        AI 善意破坏
─────────────────────              ─────────────────────────────
原因：病毒、硬件故障、误操作          原因：AI 按用户指令"正确执行"
特征：意外、突发、无预兆               特征：有意图、有逻辑、有理由
责任方：用户或系统故障                责任方：模糊（用户说了，AI做了）
防御手段：杀毒软件、RAID、备份         防御手段：⚠️ 几乎空白
用户心理：我不小心                    用户心理：我以为它懂我
```

### 1.2 三重悖论

| 悖论 | 描述 | 后果 |
|------|------|------|
| **意图悖论** | 用户说"清空测试数据"，AI 理解为"删除相关目录" | 执行结果符合 AI 逻辑，但严重偏离用户预期 |
| **能力悖论** | AI 工具越强，单次操作影响范围越大 | 能力越强 = 破坏半径越大 |
| **信任悖论** | 用户越信任 AI，越少做确认 → 越容易出事 | 信任本身成为风险因子 |

### 1.3 核心洞察

```
┌─────────────────────────────────────────────────────────┐
│                   AI 工具操作空间                        │
│                                                         │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐ │
│  │  代码生成    │    │  文件操作   │    │  网络请求   │ │
│  │  (低风险)   │    │  ⚠️ 高风险  │    │  (中风险)   │ │
│  └─────────────┘    └──────┬──────┘    └─────────────┘ │
│                            │                            │
│                    rew 聚焦此层                          │
└────────────────────────────┼────────────────────────────┘
                             ▼
                   OS 文件系统层（唯一通用介入点）
                   ──────────────────────────────
                   所有 AI 工具的文件操作
                   最终都经过这一层
```

**所有 AI 工具的破坏行为都发生在 OS 文件系统层。不需要对接每个工具，只需要在 OS 层面做防护。**

## 2. 典型事故模式（5种）

### 事故模式一：路径误判型（最常见）

```
用户指令: "清空所有测试数据，让工具处于初始化状态"
         │
         ▼
AI 推理链：测试数据 → 找到 test/ 目录 → 发现映射关系
         → 推导出 D:/Documents = 测试环境根目录
         → 执行 rm -rf D:/Documents
         │
         ▼
结果：用户 13 年文件归零（真实事故，CodeBuddy，2026-04）
```

| 触发场景 | 自然语言描述"范围"，AI 自行推导路径 |
|---------|----------------------------------|
| 高危词汇 | 清空、删除、重置、初始化、清理 |
| 影响规模 | 可能波及整个目录树 |
| 现有防御 | 几乎为零（SSD TRIM 后专业工具也无法恢复）|

### 事故模式二：批量替换型

```
用户指令: "把项目里所有的 UserInfo 改成 Profile"
AI 操作：grep -r "UserInfo" . → 找到 312 个文件 → 批量 sed 替换
        → 包含 node_modules、.git/config、系统配置文件
结果：依赖库源码被修改，项目无法启动
```

| 触发场景 | 重构、重命名、全局替换 |
|---------|----------------------|
| 影响规模 | 数十至数百文件同时变更 |
| 隐蔽性 | ⚠️ 高（问题可能延迟数天才暴露）|

### 事故模式三：配置覆盖型

```
用户指令: "帮我优化一下项目配置"
AI 操作：读取 .env → 覆盖写入新的 .env → 原有 API Keys 丢失
结果：生产环境配置丢失，服务中断
```

| 触发场景 | 配置优化、环境迁移、项目初始化 |
|---------|------------------------------|
| 高危文件 | .env、config.yaml、settings.json、docker-compose.yml |
| 恢复难度 | 极高（密钥无法重建）|

### 事故模式四：迭代覆盖型（慢性损伤）

```
Day 1: AI 修改 utils.js（"优化性能"）✓
Day 3: AI 修改 utils.js（"修复bug"）✓
Day 7: AI 修改 utils.js（"重构"）→ 发现 Day 1 之前才是正确的
结果：耗时 4 小时 git bisect，开发中断
```

| 触发场景 | 长期使用 AI 做迭代开发 |
|---------|----------------------|
| 关键痛点 | 没有"AI 操作"粒度的时间线 |

### 事故模式五：上下文污染型

```
用户处理完 ProjectA → 继续聊 ProjectB
AI 上下文混淆 → 把 ProjectA 的清理逻辑应用到 ProjectB
结果：用户完全不知情，3-7 天后才发现
```

| 隐蔽性 | ⚠️ 极高 |
|--------|--------|
| 发现时间 | 平均 3-7 天后 |

### 风险矩阵

```
发现难度
高 │  ④迭代覆盖    ⑤上下文污染
   │
中 │  ③配置覆盖
   │
低 │  ②批量替换    ①路径误判
   └────────────────────────────
      小          中          大
                        破坏规模
```

## 4. 现有解决方案缺口分析

### 对比矩阵

| 方案 | AI操作粒度 | 零学习成本 | 覆盖所有AI工具 | 终端操作保护 | 变更摘要 | 一键恢复 |
|------|-----------|-----------|--------------|------------|---------|---------|
| Git | ✗ 手动commit | ✗ 需学命令 | ✓ | ✓ | ✗ | ✗ 复杂 |
| Time Machine | ✗ 小时级 | ✓ | ✓ | ✓ | ✗ | ✗ 麻烦 |
| VSCode History | ✗ 保存级 | ✓ | ✗ | ✗ | ✗ | 单文件 |
| 云存储 | ✗ 小时级 | ✓ | ✓ | ✓ | ✗ | ✗ 麻烦 |
| **rew** | **✓** | **✓** | **✓** | **✓** | **✓** | **✓** |

### 核心缺口

```
┌─────────────────────────────────────────────────┐
│  三大结构性缺口                                   │
│                                                 │
│  缺口1：粒度                                     │
│    所有方案最细只到"保存"级                       │
│    AI 一次操作可改几十个文件，需要"操作级"粒度    │
│                                                 │
│  缺口2：感知                                     │
│    没有方案能识别"这是 AI 干的"                  │
│    因此没有异常检测，没有智能告警                 │
│                                                 │
│  缺口3：体验                                     │
│    会用的（Git）嫌麻烦                           │
│    不麻烦的（Time Machine）粒度不够              │
│    "粒度够 + 零门槛"的解法目前不存在             │
└─────────────────────────────────────────────────┘
```

## 5. 市场时机判断

### 驱动因素

| 因素 | 2024年 | 2026年（现在）|
|------|--------|--------------|
| 非技术用户使用本地AI工具 | 25% | **55%+** |
| Agent 多步骤文件操作能力 | 弱 | **强** |
| 相关事故报道/社区讨论 | 少量 | **持续爆发** |
| 直接竞品 | 无 | **窗口开放** |

### 结论

```
✅ 需求已真实爆发（事故案例持续出现，用户有明确诉求）
✅ 技术已成熟（APFS/VSS 稳定多年，Tauri 生态完善）
✅ 竞争窗口开放（大厂未重视，先发优势明显）
```
