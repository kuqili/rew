# rew — 产品需求方案（PRD）
> 版本：v1.0 | 日期：2026-04-07

---

## 1. 产品定位

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   rew 是 AI 时代的文件安全网                             │
│   让你放心地把文件交给 AI，出了事一键找回来              │
│                                                         │
│   不是 Git   ← Git 管版本，rew 管安全                   │
│   不是备份   ← 备份按小时，rew 按 AI 操作粒度            │
│   不是沙箱   ← 不阻止 AI，只在出事后兜底                 │
│                                                         │
│   心智锚点：                                             │
│   别人卷的是"让 AI 更强"                                 │
│   rew 做的是"让人敢用 AI"                               │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 2. 用户故事

### Epic 1：安心交给 AI

| ID | 角色 | 我想要 | 以便于 | 优先级 |
|----|------|--------|--------|--------|
| US-01 | Vibe Coder | 每次 AI 操作文件后自动存档 | 不需要手动 commit | P0 |
| US-02 | 非技术用户 | 看到 rew 在运行，有安心感 | 安心使用 AI 工具 | P0 |
| US-03 | 所有用户 | 完全无感，不影响 AI 工具速度 | 不会因为 rew 而放弃使用 | P0 |

### Epic 2：出事时有人告诉我

| ID | 角色 | 我想要 | 以便于 | 优先级 |
|----|------|--------|--------|--------|
| US-04 | 所有用户 | 大量文件被删/改时收到通知 | 第一时间知道出事了 | P0 |
| US-05 | 所有用户 | 通知告诉我删了多少、哪些 | 快速判断严重程度 | P1 |
| US-06 | 技术用户 | 通知里有文件变更 diff 摘要 | 不用自己查 | P2 |

### Epic 3：出事后能找回来

| ID | 角色 | 我想要 | 以便于 | 优先级 |
|----|------|--------|--------|--------|
| US-07 | 所有用户 | 看到出事前后的时间线 | 选择恢复到哪个时间点 | P0 |
| US-08 | 所有用户 | 一键恢复到选定时间点 | 不需要技术知识 | P0 |
| US-09 | 技术用户 | 选择性恢复（只恢复部分文件）| 精细化控制 | P2 |
| US-10 | 所有用户 | 恢复前预览会影响哪些文件 | 避免恢复操作本身带来问题 | P1 |

## 3. 功能优先级矩阵

### Must Have — Phase 1 必须交付

| 功能 | 描述 | 验收标准 |
|------|------|---------|
| 自动快照 | 监听目标目录，文件变更后触发 OS 原生快照 | 快照创建 < 2秒，用户无感知 |
| 异常检测 | 短时间大量文件删除/修改触发告警 | 30秒内删除 > 20 文件触发告警 |
| 系统通知 | 异常时弹出系统通知 | 通知包含：时间、文件数、目录 |
| CLI 恢复 | `rew restore` 命令行选择恢复点并恢复 | 可正常执行恢复 |
| 开机自启 | 安装后自动开机启动 | 重启后 rew 自动运行 |
| 目标目录配置 | 默认保护桌面/文档/下载，可自定义 | 首次启动 60 秒完成设置 |

### Should Have — Phase 2 交付

| 功能 | 描述 | 验收标准 |
|------|------|---------|
| 系统托盘图标 | 常驻托盘，显示运行状态 | 图标可点击显示基本状态 |
| 本地 Web UI | 浏览器可视化时间线 | 展示快照列表，可点击恢复 |
| 变更摘要 | 每个快照附带文件变更列表 | 显示：新增/修改/删除文件数 |
| 恢复预览 | 恢复前展示会影响的文件列表 | 防止误恢复 |

### Could Have — Phase 3 探索

| 功能 | 描述 | 备注 |
|------|------|------|
| Claude Code MCP 集成 | AI 操作前自动打标签 | 快照与 AI 对话关联 |
| VSCode/Cursor 插件 | 编辑器内显示时间线 | 开发者体验提升 |
| 选择性恢复 | 只恢复选定的文件/目录 | 避免全量恢复副作用 |
| 存储用量可视化 | 显示快照占用空间 | 帮助用户管理存储 |

### 明确不做

| 功能 | 原因 |
|------|------|
| 云备份/同步 | 隐私边界，增加复杂度，与定位不符 |
| 账号注册 | 增加摩擦，不符合零依赖原则 |
| 物理拦截 AI 操作 | 破坏 AI 工具体验，违背产品定位 |
| 跨设备恢复 | 超出 v1 范围 |

## 4. 核心用户体验流程

### 4.1 安装流程（目标：< 60秒）

```
下载 rew.dmg / rew-setup.exe
       │
       ▼
双击安装（< 30秒，无需额外依赖）
       │
       ▼
首次启动向导
  ┌─────────────────────────────────────┐
  │  rew 正在保护你的文件                │
  │                                     │
  │  已自动选择以下目录：                │
  │  ✅ 桌面                            │
  │  ✅ 文稿 / Documents                │
  │  ✅ 下载                            │
  │                                     │
  │  [+ 添加其他目录]                    │
  │  开机自动启动  ✅                    │
  │                                     │
  │        [开始保护]                    │
  └─────────────────────────────────────┘
       │
       ▼
窗口消失 → 系统托盘出现图标 → 首次快照自动创建

用户决策：零（默认选择已最优）
```

### 4.2 日常运行（目标：完全无感）

```
用户使用 AI 工具（Cursor / Claude Code / CodeBuddy / ...）
       │
       ▼
AI 修改文件 ← 用户完全不感知 rew 在工作
       │
       ▼
rew 后台：
  ├─ FSEvents / ReadDirectoryChangesW 检测到文件变更
  ├─ 30秒内无更多变更 → 触发快照
  ├─ 快照创建（< 1秒，CoW 增量，几乎不占新空间）
  └─ 更新托盘图标（微小动画，可选关闭）

用户感知：零
系统资源：CPU < 1%，内存 < 50MB
```

### 4.3 异常告警流程

```
rew 检测到：30秒内 47 个文件被删除
       │
       ▼
立刻触发快照（保存"出事前"状态）
       │
       ▼
系统通知弹出：
  ┌─────────────────────────────────────┐
  │ ⚠️  rew 检测到异常文件操作            │
  │                                     │
  │  47 个文件在 15 秒内被删除           │
  │  目录：~/Documents/project-x        │
  │  快照已自动保存                      │
  │                                     │
  │  [查看详情]              [先不管]    │
  └─────────────────────────────────────┘
       │
       ├─ 点击"先不管" → 通知消失，快照保留备用
       └─ 点击"查看详情" → 打开时间线界面
```

### 4.4 恢复流程

```
时间线界面（本地 Web UI，Phase 2）

  2026-04-07
  ─────────────────────────────────────────────────────
  21:16 ⚠️ 异常快照  ← 47个文件被删
  21:11 ○ 自动快照     3个文件修改        ← 选这里
  21:06 ○ 自动快照     12个文件修改
  21:01 ○ 自动快照     1个文件新增
  ─────────────────────────────────────────────────────

  选中 21:11 → [预览恢复影响]
  ┌─────────────────────────────┐
  │  将会恢复：47 个文件         │
  │  不受影响：当前新增的 2 个   │
  └─────────────────────────────┘
       │
       ▼
  [一键恢复] → 30秒完成 → 文件全部回来 ✓
```

### 4.5 那个 13 年案例，用 rew 的结果

```
21:11  ○ 自动快照（十三年文件全部在内）
21:16  ⚠️ AI 删除 D 盘大量文件
         → 系统通知立刻弹出
         → 用户点击"查看详情"
         → 选择 21:11 快照 → 恢复
         → 十三年文件，全部回来
```

## 5. 非功能需求

### 性能需求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 快照创建时间 | < 2秒 | 用户操作不受影响 |
| CPU 占用（监听态）| < 1% | 后台无感运行 |
| 内存占用 | < 50MB | 不影响 AI 工具运行 |
| 恢复时间（1000文件）| < 10秒 | 恢复体验要流畅 |
| 安装包大小 | < 10MB | 降低安装摩擦 |

### 存储策略

```
快照保留规则（分级）

当前时刻
    ├── 最近 1小时  ── 每 5分钟 1个  ── 全部保留
    ├── 最近 24小时 ── 每 1小时 1个  ── 保留 24个
    └── 最近 30天  ── 每 1天 1个    ── 保留 30个
        > 30天                       ── 自动清理

特殊规则：
  - 异常触发的快照：保留时间 × 2
  - 用户手动标记的快照：永久保留

存储占用估算（CoW 增量机制）：
  轻度使用（文档）< 100MB
  中度使用（代码）< 2GB
  重度使用（大项目）< 10GB
```

### 隐私需求

| 原则 | 实现 |
|------|------|
| 完全本地 | 快照数据不离开本机，无网络请求 |
| 无账号 | 不收集任何用户信息 |
| 无遥测（默认）| 默认不上报任何数据 |
| 零依赖 | 不需要 API Key，不需要注册 |

### 兼容性

| 平台 | 版本要求 | 文件系统要求 |
|------|---------|------------|
| macOS | ≥ 10.15 Catalina | APFS（2017年后 Mac 默认）|
| Windows | ≥ Windows 10 1903 | NTFS（VSS 依赖）|

## 6. 成功指标

### Phase 1 成功标准（3个月）

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| 安装量 | 1000+ | 下载统计 |
| 7日留存 | > 60% | 活跃天数 |
| 首次设置完成率 | > 90% | 向导完成率 |
| 快照成功率 | > 99.9% | 本地日志 |
| 崩溃率 | < 0.1% | 崩溃报告 |

### 产品健康度指标

| 指标 | 健康范围 | 含义 |
|------|---------|------|
| 恢复操作发生率 | 5-15%/月 | 需求真实存在 |
| 告警后恢复率 | > 70% | 告警有价值 |
| 用户主动恢复（无告警）| > 20% | 用户有主动意识 |
