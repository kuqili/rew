## 2. 核心模块设计

### 模块依赖图

```
FileWatcher ──▶ EventProcessor ──▶ SnapshotEngine
                      │                  │
                      ▼                  ▼
               AnomalyDetector    StorageManager
                      │
                      ▼
                   Notifier          RestoreEngine
```

### 模块职责

| 模块 | 职责 | 关键数据结构 |
|------|------|------------|
| **FileWatcher** | 监听目标目录的文件系统事件 | `FileEvent { path, kind, timestamp }` |
| **EventProcessor** | 事件去重、聚合、节流（30秒窗口）| `EventBatch { events[], window_ms }` |
| **AnomalyDetector** | 检测异常操作模式，输出告警信号 | `AnomalySignal { kind, severity, affected_files }` |
| **SnapshotEngine** | 调用 OS 原生快照 API | `Snapshot { id, timestamp, trigger, metadata }` |
| **StorageManager** | 快照生命周期管理，自动清理 | `RetentionPolicy { hourly, daily, monthly }` |
| **RestoreEngine** | 执行快照恢复操作 | `RestoreJob { snapshot_id, target_paths, dry_run }` |
| **Notifier** | 发送系统通知 | `Notification { title, body, actions[] }` |

## 3. macOS 实现方案

### 技术栈

```
用户空间进程 (rew)
├── FSEvents API        → 监听目录变更事件（OS 原生，高效）
├── tmutil CLI          → 创建/列举/删除 APFS 本地快照
└── LaunchAgent         → 开机自启（~/Library/LaunchAgents/）

OS 内核
└── APFS CoW 快照       → 增量存储，创建 < 1秒，不复制文件
```

### FSEvents 核心实现（Rust 伪代码）

```rust
use fsevent::FsEvent;

pub struct MacOSWatcher {
    paths: Vec<PathBuf>,
    tx: Sender<FileEvent>,
}

impl MacOSWatcher {
    pub fn start(&self) {
        let mut observer = FsEvent::new(self.paths.clone());
        observer.observe(|events| {
            for event in events {
                // 过滤系统文件、.DS_Store 等噪音
                if !self.should_ignore(&event.path) {
                    self.tx.send(FileEvent {
                        path: event.path,
                        kind: map_fsevent_kind(event.flag),
                        timestamp: Utc::now(),
                    });
                }
            }
        });
    }
}
```

### APFS 快照操作

```bash
# 快照创建（< 1秒）
tmutil localsnapshot

# 列举所有快照
tmutil listlocalsnapshots /

# 挂载快照（用于恢复预览）
tmutil mount [snapshot-name]

# 执行恢复
tmutil restore -s [snapshot-name] [source-path] [dest-path]

# 删除快照（存储管理）
tmutil deletelocalsnapshots [snapshot-date]
```

### LaunchAgent 配置


## 4. Windows 实现方案

### 技术栈

```
用户空间进程 (rew.exe)
├── ReadDirectoryChangesW API  → 监听目录变更事件
├── VSS (Volume Shadow Copy)   → 创建/管理卷影快照
└── Task Scheduler             → 开机自启

OS 内核
└── NTFS + VSS CoW 快照        → 增量存储
```

### ReadDirectoryChangesW 核心实现（Rust 伪代码）

```rust
use winapi::um::fileapi::*;
use winapi::um::winnt::*;

impl WindowsWatcher {
    pub fn watch(&mut self, path: &Path) {
        let h = CreateFileW(
            path, FILE_LIST_DIRECTORY,
            FILE_SHARE_READ | FILE_SHARE_WRITE,
            null_mut(), OPEN_EXISTING,
            FILE_FLAG_BACKUP_SEMANTICS | FILE_FLAG_OVERLAPPED,
            null_mut()
        );

        loop {
            ReadDirectoryChangesW(
                h, self.buffer.as_mut_ptr() as *mut _,
                self.buffer.len() as u32,
                TRUE,  // 递归监听子目录
                FILE_NOTIFY_CHANGE_FILE_NAME |
                FILE_NOTIFY_CHANGE_DIR_NAME  |
                FILE_NOTIFY_CHANGE_LAST_WRITE,
                &mut bytes_returned, &mut overlapped, None
            );
            self.process_events();
        }
    }
}
```

### VSS 快照操作（Rust 伪代码）

```rust
use windows::Win32::Storage::Vss::*;

impl VssSnapshot {
    pub fn create(&mut self, volume: &str) -> Result<Uuid> {
        self.backup_components.InitializeForBackup(None)?;
        self.backup_components.SetBackupState(false, false, VSS_BT_COPY, false)?;
        let snapshot_id = self.backup_components.AddToSnapshotSet(volume, None)?;
        self.backup_components.DoSnapshotSet(None)?;
        Ok(snapshot_id)
    }

    pub fn restore(&self, snapshot_id: Uuid, dest: &Path) -> Result<()> {
        let shadow_path = self.get_shadow_path(snapshot_id)?;
        copy_dir(shadow_path, dest)
    }
}
```

### 双平台对比

| 维度 | macOS (APFS) | Windows (VSS) |
|------|-------------|--------------|
| 快照创建耗时 | < 1秒 | 2-10秒 |
| 增量存储 | ✓ CoW | ✓ CoW |
| 挂载预览 | 需 SIP 权限 | 需管理员 |
| 快照数量限制 | 无硬性限制 | 建议 < 512 |
| API 稳定性 | 高 | 高（Win7+）|
| 权限要求 | 普通用户 | **需要管理员** |

> **Windows 权限策略**：安装时请求一次管理员权限，注册为系统服务，后续完全无感运行。

## 5. 异常检测算法

### 规则引擎

```
事件流
  │
  ▼
┌─────────────────────────────────────────────────────┐
│                  滑动时间窗口（30秒）                 │
│                                                     │
│  RULE-01: DELETE 数量 > 20          → HIGH          │
│  RULE-02: DELETE 数量 > 5           → MEDIUM        │
│  RULE-03: 删除总大小 > 100MB        → HIGH          │
│  RULE-04: MODIFY 数量 > 50          → MEDIUM        │
│  RULE-05: 被监听根目录被删除         → CRITICAL      │
│  RULE-06: .env/.gitconfig 等被修改  → HIGH          │
│  RULE-07: node_modules 外 > 30 修改 → MEDIUM        │
│                                                     │
└─────────────────────────────────────────────────────┘
  │
  ├─ CRITICAL → 立刻快照 + 立刻通知
  ├─ HIGH     → 立刻快照 + 立刻通知
  └─ MEDIUM   → 立刻快照 + 延迟通知（聚合后通知）
```

### 规则详细定义

| 规则ID | 触发条件 | 级别 | 说明 |
|--------|---------|------|------|
| RULE-01 | 30秒内删除文件数 > 20 | HIGH | 最常见 AI 误删模式 |
| RULE-02 | 30秒内删除文件数 > 5 | MEDIUM | 小批量删除预警 |
| RULE-03 | 30秒内删除总大小 > 100MB | HIGH | 按大小检测 |
| RULE-04 | 30秒内修改文件数 > 50 | MEDIUM | 大范围修改 |
| RULE-05 | 监听根目录被删除 | CRITICAL | 最危险操作 |
| RULE-06 | .env/.gitconfig 等被修改 | HIGH | 配置覆盖检测 |
| RULE-07 | node_modules 外 30秒内 > 30 修改 | MEDIUM | 排除安装噪音 |

### 噪音过滤列表

```
路径过滤（不触发检测）：
  **/node_modules/**
  **/.git/**
  **/target/**
  **/__pycache__/**
  **/.DS_Store
  **/Thumbs.db

操作过滤：
  检测到 package.json 变化 → 暂停 node_modules 检测 30秒
  检测到 .git/HEAD 变化   → 暂停检测 10秒
```

## 6. 存储管理策略

### 数据目录结构

```
~/.rew/                          （macOS）
%APPDATA%\rew\                   （Windows）
│
├── snapshots.db                 ← SQLite，快照元数据
│   ├── id
│   ├── timestamp
│   ├── trigger                  (manual / auto / anomaly)
│   ├── os_snapshot_ref          ← 指向 OS 原生快照 ID
│   ├── files_added
│   ├── files_modified
│   └── files_deleted
│
├── config.toml                  ← 用户配置（保护目录列表等）
└── rew.log                      ← 运行日志（按天切割）
```

> **关键设计**：rew 本身不存储文件副本，只存储元数据 + OS 快照引用。
> 实际文件数据由 APFS/VSS 的 CoW 机制管理，存储效率极高。

### 保留策略

```
触发时机：每次创建新快照后执行清理

┌────────────────┬──────────────┬────────────────────┐
│  时间范围       │  保留粒度    │  最大保留数         │
├────────────────┼──────────────┼────────────────────┤
│  0 - 1小时     │  全部保留    │  无限制             │
│  1 - 24小时    │  每小时最新  │  24个               │
│  1 - 30天      │  每天最新    │  30个               │
│  > 30天        │  全部删除    │  0                  │
└────────────────┴──────────────┴────────────────────┘

特殊规则：
  - anomaly 触发的快照：保留时间 × 2
  - 用户手动标记的快照：永久保留，不参与自动清理
```

### 存储占用估算

| 场景 | 日变更量 | 快照数 | 预估占用 |
|------|---------|--------|---------|
| 轻度（文档）| < 1MB | 66个 | < 100MB |
| 中度（代码）| < 50MB | 66个 | < 2GB |
| 重度（大项目）| < 500MB | 66个 | < 10GB |

## 7. 技术风险与对策

| 风险 | 概率 | 影响 | 对策 |
|------|------|------|------|
| 文件事件噪音过多 | 高 | 存储浪费、性能 | 30秒窗口聚合 + 路径过滤 |
| 磁盘空间被快照占满 | 中 | 用户体验差 | 超过 10GB 告警 + 自动清理 |
| Windows VSS 权限被拒 | 中 | 功能不可用 | 安装时一次性申请，注册系统服务 |
| VSS 在某些配置下不可用 | 低 | 降级 | Fallback：文件级增量复制 |
| rew 进程崩溃 | 低 | 保护失效 | 守护进程监控 + 崩溃自动重启 |
| APFS 快照被 OS 自动删除 | 低 | 快照丢失 | 监听删除事件，更新 DB，通知用户 |
| 与 Time Machine 冲突 | 低 | 快照混乱 | 独立快照名前缀，互不干扰 |
