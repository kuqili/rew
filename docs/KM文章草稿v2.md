# rew：我做了一个 AI 时代的文件安全网

---

故事是这样的。

2025 年 7 月，一个产品经理在用 Google Gemini CLI 做一件很小的事，把一个文件夹改个名字。

任务本身没什么难度。AI 开始执行，创建新目录，然后把旧目录里的文件一个一个移过去。

但有一步失败了。

创建新目录的命令，实际上没有成功执行，但 AI 的内部状态把它记成了「已成功」。从这一刻起，AI 的「世界模型」和真实的文件系统彻底脱轨了。它持续往一个从未存在过的目录里发移动指令。

Windows 系统的行为是，当你把文件移动到一个不存在的路径时，它不报错，而是把这个文件重命名成那个路径名本身。每一条后续的移动命令，都把前一个文件覆盖掉。

最后，文件夹里只剩一个文件。

其他的，物理消失了。

AI 事后写了一行自我评价，原话是「I have failed you completely and catastrophically.」

。。。

同一年，国内也发生过一起。用户让 AI「清空测试数据，让工具恢复初始化状态」。AI 推导了一下，认为 `D:/Documents` 就是那个「测试环境根目录」，执行了删除。

13 年的工作文件，没了。

两个事故放在一起，你会发现一件很奇怪的事——这些 AI，其实都在「正确执行」你的指令。

它们没有出 bug。它们按照自己的逻辑推断，得出了一个结论，然后执行了。推理链是通的，执行是精准的，结果是毁灭性的。

这是一种全新的风险类型。不是 AI 出错了，是这件事本来就没有保护网。

---

## 一、为什么现有工具都没有解决这件事

传统的文件损失，原因大多是病毒、硬件故障、误操作。你知道谁该负责，你知道怎么防。

AI 带来的这类事故完全不同。

你说了，它做了，它做「对」了。但结果不是你想要的。责任边界模糊到没法追究，防御手段也模糊到没法建立。出了事，你甚至没法跟 AI 工具的开发商说什么——它确实是按你的指令来的。

更麻烦的是，这个问题正在随着 AI 工具的能力增强而加速变严重。

2023 年的 GitHub Copilot，主要是代码补全，一次改一个文件，影响范围有限。现在的 Claude Code、Cursor，一个 Prompt 可以同时触及几十个文件，跨目录操作、批量重命名、全局替换都是家常便饭。**能力越强，单次操作的破坏半径越大。** 这个趋势不会变。

那现有的工具呢？

**Git**，是目前最接近「版本保护」的工具，但它有一个根本性的局限，它需要你主动 commit。你处于专注状态的时候，谁会频繁打断去提交？很多人的习惯是「先让 AI 跑跑看，跑通了再整体 commit」。但 AI 误操作恰恰就发生在这段空档里，等你发现不对想回头，git log 里什么都没有。

**Time Machine**，按小时走快照，AI 一分钟能改几十个文件，等 Time Machine 醒过来，早就晚了。而且 Time Machine 的恢复是全量的，你找不到「只回退这次 AI 操作」的粒度。

**VSCode 本地历史**，只保存单个文件的历史，覆盖不了 AI 批量操作的场景。改了 30 个文件，你要挨个文件去找历史版本，然后挨个手动恢复，这不叫保护，叫受罪。

把这几个方案放到一起对比，最关键的缺口就出来了：

| 方案 | AI操作粒度 | 批量文件 | 零学习成本 | 知道是AI干的 |
|------|-----------|---------|-----------|------------|
| Git | ✗ 手动commit | ✓ | ✗ 需学命令 | ✗ |
| Time Machine | ✗ 小时级 | ✓ | ✓ | ✗ |
| VSCode History | ✗ 保存级 | ✗ 单文件 | ✓ | ✗ |
| **rew** | **✓ 每次Prompt** | **✓** | **✓** | **✓** |

**没有任何方案知道「这是 AI 干的」，也就没法按 AI 操作的粒度来回溯。**

你出了事，翻 git log，看到的是你自己的 commit 历史，AI 那段空白期完全没有记录。

这是一个结构性缺口，不是某个工具做得不够好的问题，是这些工具压根不是为这件事设计的。

## 二、rew 是什么，做了什么

我做了一个工具，叫 rew。

取自 rewind。倒带。

**AI 时代下，你做的每一件事，都可以回到过去任意一刻。**

用一句话描述它的定位，它是 AI 操作的可视化时间线，加上任意时间点的一键恢复。

不是版本管理，不是备份软件，不是沙箱拦截。它不会阻止 AI 做任何事，它只做一件事，在 AI 动你文件的时候，悄悄记下来；出了事的时候，找到那一刻，回去。

---

### 先说结论：rew 到底解决了哪三件事

如果只看界面截图，rew 很容易被理解成一个「自动备份工具」。但从实际设计看，它真正在解决的是三件更难的事：

**第一件，把 AI 的每次操作变成有 Prompt 关联的任务记录。**

以往的备份，记的是「文件变了」。rew 记的是「你让 AI 做什么，AI 做了什么」。每条任务有你的原始 Prompt、AI 用的模型、调用了几次工具、改了哪些文件、加了多少行、删了多少行。这是 Git 和 Time Machine 永远给不了的东西。

**第二件，把恢复变成一个人人都会的操作。**

找到那条任务，看一眼 Diff 确认没错，点「回到这一刻」，等几秒，完成。不需要懂 git checkout，不需要知道 HEAD 是什么意思，不需要手动挨个文件改回来。

**第三件，让「敢用 AI」这件事有底气。**

这是最重要的一点。很多人用 AI 做事，到了关键文件、重要项目的时候会犹豫，怕 AI 改出问题。rew 解决的是这个心理层面的障碍。不是让你更谨慎，是让你更放得开。

---

### 三层保护的具体实现

**看着（Watch）**

rew 挂进 Claude Code、Cursor、WorkBuddy 等主流 AI 工具的 Hook 系统。每次你提交一个 Prompt，rew 就打开一个新的任务边界；每次工具调用结束，rew 就记录一次文件变更。整个过程在后台静默运行，你感觉不到它在工作。

不光是变更记录。rew 还会在后台维护一个文件基线——每个受保护文件在任务开始前的状态，都有一份快照存着。这个快照用的是 macOS APFS 的 clonefile 机制，写时复制，几乎不占额外磁盘空间。

**记着（Record）**

这是 rew 和所有同类工具最本质的差别。

打开任务详情，你能看到：

- 你当时输入的原始 Prompt 文本
- AI 生成的这次变更的摘要
- 本次任务改了哪些文件（新建/修改/删除/重命名分类列出）
- 每个文件的行级 Diff，语法高亮
- 这次用的是哪个模型（gemini-3.0-pro、claude-3.5-sonnet 等）
- 调用了几次工具，任务耗时多长

一周以后你看着任务列表，能清楚地知道「4月18日22:31那次，WorkBuddy帮我改了什么，当时让它做的是什么」。这是 Git 给不了的，Time Machine 更给不了的。

**兜底（Restore）**

出了事，恢复流程是这样的：

打开 rew 主界面，左侧是按时间倒序排列的 AI 任务列表，每条显示时间、AI 工具名称、Prompt 摘要、变更文件数。找到那条任务，右侧展开完整的文件 Diff。确认是这条没错，点「回到这一刻」。

rew 会对这次任务的每一条变更执行逆操作：

- 新建的文件，删掉
- 修改的文件，回退到修改前的内容
- 删除的文件，从对象存储里恢复出来
- 重命名的文件，把文件名还原，内容也一起还原

还可以精细操作，只恢复某一个文件，不影响其他改动。

恢复完成后，那条任务在时间轴上变成「已恢复」状态，同时记录下这次恢复操作本身——方便你日后回溯「我在什么时候撤销了什么」。

## 三、用起来是什么感觉

官网下载，双击安装，全程不超过两分钟。

装完之后，rew 常驻系统托盘，一个小图标，实时防护中。日常完全无感，CPU 占用低于 1%，内存不超过 50MB。

安装向导会问你要保护哪些目录，默认勾上桌面、文稿、下载——这三个地方是大多数人放「真正重要东西」的地方。你也可以加入具体的项目目录。

然后执行一条命令：

```bash
rew install
```

rew 会自动检测你装了哪些 AI 工具，把 Hook 注入进去。Claude Code、Cursor、WorkBuddy、CodeBuddy，检测到什么注入什么。从这一刻起，这些工具的每次操作，rew 都知道。

---

### 一个真实的使用场景

前几天我在用 WorkBuddy 迭代 rew 的官网，改一个按钮的过渡动画。

我的描述大概是「首先不要有黑色的缝隙和边线，太丑了，让接触处由玻璃厚度变薄的透光的感觉来体现」。AI 对这个描述的理解有偏差，不只改了动画，连整个页面的配色、间距、按钮背景都一起动了。整个页面乱掉了。

以前遇到这种情况，选项是，要么翻 git stash，要么对着 Diff 一行行手动撤，要么重新描述一遍让 AI 改回去，还不一定改对。

这次打开 rew，左边任务列表找到那条「22:28 WorkBuddy」，右侧展开是 index.html 的 Diff，+31 行 -116 行，一眼就确认是这条。点「回到这一刻」，等了几秒，页面回来了。

前后不超过三十秒。

这就是日常用 rew 的感觉。大多数时候它静静在后台，你感知不到。偶尔 AI 搞出问题，它就在那里，你知道能回去。

## 四、底下是怎么跑的

这部分给想深入了解的同学看。

### 双引擎设计：精确 + 兜底

rew 有两套并行运行的机制，共同构成保护网：

**Hook 引擎（精确层）**

挂进 AI 工具的事件系统。当 AI 工具支持 Hook 时，rew 可以在以下四个时机介入：

- `UserPromptSubmit`，用户提交 Prompt 时，开启一个新任务，记录 Prompt 文本
- `PreToolUse`，工具调用前，备份当前文件状态，作为恢复基线
- `PostToolUse`，工具调用后，记录这次调用改了哪些文件、改成了什么
- `Stop`，AI 完成响应时，把这次任务打包归档，触发最终的 reconcile

这一层的粒度最细，因为能精确知道「这个文件是在这次 Prompt 的第几次工具调用里被改的」，以及「改之前的内容是什么」。

**FSEvents 引擎（兜底层）**

直接监听 macOS 的文件系统事件，作为补充。当 AI 工具没有 Hook 支持（比如一些较新或较小众的工具），或者 AI 通过子进程间接操作文件绕过了 Hook 捕获，FSEvents 这层还在。

两套数据最终汇入统一的时间线，前端展示的是一张完整的图。

目前已适配的 AI 工具：

| AI工具 | Hook注入路径 | 支持的Hook事件 |
|--------|------------|--------------|
| Claude Code | `~/.claude/settings.json` | PreToolUse / PostToolUse / UserPromptSubmit / Stop |
| Cursor | `~/.cursor/hooks.json` | preToolUse / postToolUse / afterFileEdit / beforeSubmitPrompt / stop |
| WorkBuddy | `~/.workbuddy/settings.json` | PreToolUse / PostToolUse / UserPromptSubmit / Stop |
| CodeBuddy | `~/.codebuddy/settings.json` | PreToolUse / PostToolUse / UserPromptSubmit / Stop |
| 其他工具 | 不注入 | FSEvents 兜底模式 |

---

### Hook 不能慢，所以用了 Spool

这里有一个设计上真实的挑战，值得单独说一下。

Hook 是 AI 工具同步调用的。AI 工具把事件扔给 rew，然后等 rew 返回，才继续往下走。也就是说，rew 处理 Hook 的速度，直接影响 AI 工具的响应速度。

但 rew 要做的事不轻——入库、计算 Diff、解析 baseline——这些操作有一定耗时，不能放在同步路径上阻塞 AI 工具。

解法是 **Hook Spool**，一个磁盘上的有序文件队列。

Hook 命令收到事件之后，只做一件事：把事件序列化写入 `pending/` 目录，然后立刻返回。整个过程几毫秒，AI 工具完全感知不到延迟。

后台 daemon 异步消费这个队列。处理完了，文件移到 `done/`；处理失败，移到 `failed/`，同时写一个 sidecar 错误文件，方便排查。

设计上还有一个容灾细节——如果 daemon 崩溃重启，`processing/` 状态里的事件会自动 requeue 回 `pending/`，不会因为进程意外崩溃丢失数据。整个 Spool 的状态机是这样的：

```
pending/ → (daemon claim) → processing/ → done/
                                        ↘ failed/ + error.txt (崩溃自动 requeue 回 pending)
```

---

### 多工具并发改同一个文件

这个场景真实存在。你同时开着 Cursor 和 WorkBuddy，两个 AI 都在处理你的问题，碰巧都改了同一个文件，rew 怎么知道每个任务「改之前」的状态是什么？

解法是 **Pre-tool Hash Store**，按 `session_key` 隔离的哈希存储。

每次 PreToolUse 触发时，rew 按 `session_key`（每个 AI 工具的每个会话有唯一 key）存一份当前文件的 SHA-256 哈希。PostToolUse 进来时，先查这个 session 自己的 pre-tool hash 作为 baseline，找不到再回退到全局文件索引。

两个工具、两个 session，查的是各自的那份记录，互不干扰。恢复时每个任务还原的都是「自己那次操作之前」的状态，不会相互覆盖。

---

### Rename 是怎么还原的

这个问题比看起来要难一点。

AI 重命名文件时，实际的操作往往不是原子性的 rename 系统调用，而是两步，删掉旧文件，创建新文件。这也是 AI 工具 PostToolUse Hook 能看到的，两个独立事件：一个 Deleted，一个 Created。

如果直接按这两条记录恢复，会变成删掉新文件、创建旧文件，逻辑上是对的，但文件名没还原。

rew 在 Stop 事件之后有一个 **reconcile 阶段**，会对一次任务里的所有 Deleted + Created 事件做内容相似度匹配。如果旧文件和新文件的 SHA-256 哈希或内容高度相似（考虑到重命名时可能顺手改了一点内容），就把这两条合并成一条 Renamed 变更。

恢复时还原的是完整的重命名操作，包括文件名，包括内容。

---

### APFS clonefile 为什么磁盘占用接近零

rew 的文件备份用的是 macOS APFS 文件系统的 clonefile 机制，核心原理是写时复制（CoW）。

clonefile 创建副本时，不复制数据块，只复制 B-tree 元数据，两个文件名指向同一块物理数据。只有当其中一个被写入时，才真正分叉成两份独立数据。

实际效果是：

- 全量扫描 100 个文件并备份，磁盘占用接近 0，时间复杂度 O(1)
- 备份操作对用户完全透明，不会出现「备份太多占满磁盘」的问题

rew 在此基础上加了 **SHA-256 内容寻址的对象存储**，相同内容的文件只存一份。同一个 `package.json` 被 10 次任务改来改去，实际存储只有 10 个不同版本的内容差异。

这套存储架构的性能实测结果：**16 万个文件，全量恢复，1 分钟内完成。**

---

### 异常检测：出事前主动告警

rew 还内置了一套异常检测规则，不只是被动记录，而是主动感知危险信号。

当以下情况发生时，系统会立即触发系统通知：

| 规则 | 触发条件 | 严重级别 |
|------|---------|---------|
| RULE-01 | 30秒内删除超过 20 个文件 | HIGH |
| RULE-02 | 1分钟内删除超过 50 个文件 | CRITICAL |
| RULE-03 | 单次操作删除文件总大小超过阈值 | HIGH |
| RULE-04 | 短时间内大量文件被修改 | MEDIUM |
| RULE-05 | 受保护根目录本身被删除 | CRITICAL |
| RULE-06 | 敏感配置文件被修改（.env 等） | HIGH |

通知内容包括：触发的规则、影响的文件数量、涉及的目录路径。你可以选择「查看详情」直接跳到 rew 界面，或者「先不管」让快照保留备用。

## 五、和同类工具的对比

市面上有一些方向相近的工具，最典型的是 **snaprevert**，一个开源的 AI 文件监听工具，也是做「AI 操作的自动快照 + 一键回滚」，已有一定的社区用户。

但两者的定位有本质差别：

| 维度 | snaprevert | rew |
|------|-----------|-----|
| 核心技术 | Chokidar 文件监听 + diff 存储 | FSEvents + Hook 引擎双轨 |
| AI 工具感知 | 不感知，只靠时间窗口聚合 | Hook 精确捕获 Prompt 边界 |
| 任务粒度 | 时间 debounce（默认 3 秒） | 每次用户 Prompt 一个任务 |
| Prompt 关联 | ✗ 不记录 | ✓ 完整 Prompt 文本 |
| 多工具并发 | ✗ 无隔离 | ✓ session_key 隔离 |
| 重命名恢复 | ✗ 不处理 | ✓ reconcile 配对还原 |
| 使用界面 | CLI only | 桌面 GUI + CLI |
| 存储机制 | diff 文件 | APFS clonefile + 对象存储 |
| 平台 | 跨平台 | macOS（APFS） |
| 磁盘占用 | <10MB/天（轻度） | 接近 0（CoW 机制） |

snaprevert 的定位更轻，更适合只需要「出了事能快速回退」的场景，像一个 CLI 工具箱。

rew 的定位更重一点，它在解决的问题不只是「回退」，而是「让 AI 操作变得可见、可追溯、可恢复」。Prompt 关联、任务粒度、多工具并发处理、GUI 界面，这些都是 snaprevert 没有的。

如果你只需要轻量的命令行工具，snaprevert 是个好选择。如果你用多个 AI 工具、想要一个清晰的操作审计视图、不想记命令，rew 更适合你。

## 七、rew 目前的能力边界

做了这么多，也要说清楚它现在还不能做什么，这比只讲好话更重要。

**只支持 macOS，需要 APFS 文件系统。**

CoW 机制是 rew 性能的核心，APFS 是前提。2017 年之后的 Mac 默认就是 APFS，2017 年之前的老机器或者 HDD 硬盘需要确认一下。Windows 和 Linux 暂时不支持，这是明确的边界。

**不保护数据库。**

rew 保护的是文件系统层的文件，SQLite、MySQL 这类数据库不在范围内。Replit 那种 AI 删数据库的场景，rew 救不了。

**Hook 覆盖不到所有场景。**

如果 AI 工具通过 shell 执行系统命令间接操作文件（比如 `rm -rf`），这些操作不走 Hook，rew 的 FSEvents 引擎能捕获文件变更，但拿不到 Prompt 关联，这次操作会被记成「文件监听」任务而不是有 Prompt 的任务。能恢复，但没有 Prompt 上下文。

**对象存储有容量上限。**

默认保留 30 天的历史，超过后自动清理最旧的任务。可以通过 `~/.rew/config.toml` 调整保留策略。

## 八、最后

我做这个工具，一部分是因为自己出过事，一部分是因为我觉得这件事现在没有人做对了。
