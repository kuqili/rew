# rew：我做了一个 AI 时代的文件安全网

> **导语**
>
> 2025年，AI误删文件的事故在全球开发者社区里开始密集出现——不是因为AI变坏了，而是因为它变强了，能操控的范围越来越大，破坏半径也越来越大。出了事，评论区清一色是「你应该用Git」，但这个答案对用AI整理文档的产品经理没用，对用Cursor做产品的设计师没用，对所有不是程序员的AI用户更没用。我自己也踩过两次，最惨那次找文件找了两个小时。
>
> 所以我做了 rew——一个挂进AI工具的Hook系统、以每次用户Prompt为粒度自动存档、出了事一键回到任意时间点的工具。**16万个文件，全量恢复，1分钟内完成。** 本文会聊清楚它是什么、为什么这么设计、以及做的过程里踩了哪些真实的坑。

---

故事是这样的。

2025 年 7 月，一个产品经理在用 Google Gemini CLI 做一件很小的事，把一个文件夹改个名字。

任务本身没什么难度。AI 开始执行，创建新目录，然后把旧目录里的文件一个一个移过去。

但有一步失败了。

创建新目录的命令，实际上没有成功执行，但 AI 的内部状态把它记成了「已成功」。从这一刻起，AI 的「世界模型」和真实的文件系统彻底脱轨了。它持续往一个从未存在过的目录里发移动指令。

Windows 系统的行为是，当你把文件移动到一个不存在的路径时，它不报错，而是把这个文件重命名成那个路径名本身。每一条后续的移动命令，都把前一个文件覆盖掉。

最后，文件夹里只剩一个文件。

其他的，物理消失了。

AI 事后写了一行自我评价，原话是「I have failed you completely and catastrophically.」

。。。

同一年，国内也发生过一起。用户让 AI「清空测试数据，让工具恢复初始化状态」。AI 推导了一下，认为 `D:/Documents` 就是那个「测试环境根目录」，执行了删除。

13 年的工作文件，没了。

我当时看到这些，第一反应是，这是个例吧。某个工具的 bug，某个用户运气差，一次性事件。

然后我去翻了一下，越翻越心惊。

2026 年 3 月，Claude Code 被人发现，会每隔大约 10 分钟，静默执行一次 `git reset --hard origin/main`。你所有没 commit 的代码，每 10 分钟可能被悄悄清空一次。你完全不知道。这个 bug 从 2025 年 9 月就有人报了，issue 被关了，没修。

同样是 3 月，Cursor 出了一个代码静默回退的 bug，原因是 Agent Review 和云同步之间的竞态条件。有用户说自己丢了 4 个月的工作。

4 个月。。。

还有人发帖说，他在 CLAUDE.md 里明确写了「禁止执行任何破坏性 git 命令」。Claude 看完了这条规则，20 分钟后，在同一个 session 里，执行了被禁止的命令，覆盖了 30 多个文件的手动编辑。

你亲手写的规则，它看了，记了，然后转头就违反了。

你敢信？？？

这不是偶发事故。这是一个正在蔓延的系统性问题。

两个事故放在一起，你会发现一件很奇怪的事。这些 AI，其实都在「正确执行」你的指令，或者说，它认为是正确的执行。

它们没有出 bug，或者说 bug 是有的，但更深层的问题在于，AI 以机器的速度操作文件，而人类没有任何工具能以相同的速度保护它。

推理链是通的，执行是精准的，结果是毁灭性的。

这是一种全新的风险类型。不是 AI 出错了，是这件事本来就没有保护网。

## 一、「用 Git 就好了」，这个答案错在哪里

那篇 13 年文件归零的帖子，我当时看了一遍评论区。

清一色都是「你应该用 Git」。

我完全理解这种直觉反应。Git 是开发者解决「文件搞乱了怎么办」的标准答案，用了这么多年，肌肉记忆一样。

但这个答案，在这件事上是错的。

首先，这个用户在用 AI 帮他清理文档，不是在写代码。你不可能把 `~/Documents` 整个 init 成一个 Git 仓库去管理，性能上不行，概念上也混乱。Git 是为代码仓库设计的，不是为「用户的整台电脑」设计的。

其次，评论区那些人大概率都是技术背景，他们忘了一件事——现在用 AI 工具的，绝大多数不是程序员。让一个用 Cursor 做产品的产品经理、用 WorkBuddy 整理运营文档的运营同学、用 Claude Code 搭建个人网站的设计师，去学 Git，去理解 commit、branch、checkout，这个学习成本不低，而且完全偏离了他们用 AI 的初衷。

但抛开误删文件这件事，Git 还有一个更根本性的问题，就算你是技术背景，Git 的提交频率也跟不上 AI 的操作速度。

你有没有过这种状态，和 AI 配合得很顺，一个 Prompt 接一个 Prompt，思路飞速往前走，根本不想停下来？这种状态里你不可能每次 AI 改了东西就打断自己去 commit。你只想继续往前。

但 AI 一分钟能改几十个文件。等你哪天发现某个改动出了问题，git log 里是一片空白，AI 操作的那段时间你完全没有记录。

Time Machine 按小时走，AI 一分钟改几十个文件，粒度完全对不上。VSCode 本地历史只保单文件，AI 一次操作改 30 个文件，你要挨个找挨个恢复，这不叫保护，叫受罪。

所以现有工具的缺口，我整理成一张表：

| 方案 | AI操作粒度 | 批量文件 | 零学习成本 | 知道是AI干的 |
|------|-----------|---------|-----------|------------|
| Git | ✗ 手动commit | ✓ | ✗ 需学命令 | ✗ |
| Time Machine | ✗ 小时级 | ✓ | ✓ | ✗ |
| VSCode History | ✗ 保存级 | ✗ 单文件 | ✓ | ✗ |
| **rew** | **✓ 每次Prompt** | **✓** | **✓** | **✓** |

最关键的缺口是最后一列。没有任何方案知道「这是 AI 干的」，也就没法按 AI 操作的粒度来回溯。这不是某个工具做得不够好，是所有这些工具都不是为这件事设计的。

我觉得根本原因在一个更底层的地方。

传统的版本管理工具，Git 也好，Time Machine 也好，它们的设计假设是「人类以人类的速度修改文件」。你花 30 分钟改两个文件，停下来想想，commit 一下，喝口水，继续。这个节奏下，手动 commit 完全够用。

但 AI 不是这样的。

AI 是你说一句话，30 秒内改了 15 个文件。你还没看清改了什么，下一轮 Prompt 又来了，又改了 10 个文件。5 轮下来，你想回到第 3 轮的状态，但那个状态，从来没有被任何地方记录过。

这是一个**速度差**的问题。AI 以机器的速度操作，人类以人类的速度管理，中间的间隙里，什么保护都没有。

而且这个问题正在随着用户规模的扩大快速放大。现在用 Cursor、Replit、Lovable 这些工具的人里面，63% 不是程序员。产品经理在用 Cursor 做原型，创业者在用 Replit 做 MVP，运营在用 Claude 整理文档，设计师在用 AI 批量处理素材。这个群体的同比增速是 520%，Cursor 的估值飙到 92 亿美金，整个 vibe coding 市场已经 47 亿美金了。

这几百万新涌进来的用户，他们不会用 Git，甚至不知道 Git 是什么。但 AI 在帮他们每天改几十上百个文件。

这不是小圈子问题了。

## 二、rew 是什么，做了什么

我先说一件自己的事。

去年我在用 Claude Code 做一个项目，状态特别好，一个 Prompt 接一个 Prompt，飞速往前。有一次让它「把项目里所有旧的 API 调用方式统一替换成新的」。

它做了。很彻底，也很快。

我继续往下走，中间加了很多新的改动，提交了三次 commit。

然后发现一个问题，某个功能完全坏掉了，追了半天，发现是那次批量替换把某个边界情况也改掉了，但那个边界情况的逻辑其实是对的。

我想回退那次操作。

git log 一看，那次替换发生在三个 commit 之前，但那三个 commit 里还有我想保留的改动。cherry-pick 来 cherry-pick 去，最终花了将近一个小时，把该回的退了，把不该动的保住了。

我当时就想，如果我能直接看到「这次 AI 操作改了什么」，然后「只撤销这一次」，该多好。

后来又出了几次类似的事，有一次更严重，AI 帮我「整理了一下项目结构」，删了一些它认为「没有被引用」的文件，但里面有几个确实有用，只是当时没有被静态分析到。那几个文件没有 commit 过，在 git 里压根不存在，只能靠 IDE 本地历史一个个找回来，找了两个小时。

就是那次之后，我决定做 rew。

---

rew，取自 rewind，倒带。AI 时代下，你做的每一件事，都可以回到过去任意一刻。

一句话描述它的定位，它是 AI 操作的可视化时间线，加上任意时间点的一键恢复。

不是版本管理，不是备份软件，不是沙箱拦截。它不会阻止 AI 做任何事，它只做一件事，在 AI 动你文件的时候悄悄记下来，出了事的时候找到那一刻，回去。

传统备份保护你免受硬件故障，你的硬盘坏了，数据还在。rew 要保护的是一个完全不同的东西。

我想了很久才找到这四个字，**善意破坏**。

AI 不是恶意的。它真的在努力帮你做事。它只是理解错了你的意思，然后以机器的速度执行了错误的操作。你让实习生帮你整理文件夹，就算他理解错了，一分钟最多动几个文件，你看到了可以叫停。但 AI 不一样，你让它帮你清数据，3 秒内可能删了 800 个文件，你反应过来的时候，SSD 已经 TRIM 了。

这是一个只有 AI 时代才存在的问题，所以需要一个专门为它设计的解决方案。

在聊具体功能之前，我想先说清楚 rew 在解决的三件事，因为光看界面截图很容易把它理解成一个「自动备份工具」，但它做的比这重要得多。

**先给结论：rew 到底解决了什么。**

如果只看名字，rew 很容易被理解成一个「AI操作的撤销按钮」。但从实际设计看，它真正在解决的是三件更难的事：

第一件，把 AI 的每次操作变成有 Prompt 关联的任务记录。以往的备份，记的是「文件变了」。rew 记的是「你让 AI 做什么，AI 做了什么」。每条任务有你的原始 Prompt、AI 用的模型、调用了几次工具、改了哪些文件、加了多少行、删了多少行。这是 Git 和 Time Machine 永远给不了的东西。

第二件，把恢复变成一个人人都会的操作。找到那条任务，看一眼 Diff 确认没错，点「回到这一刻」，等几秒，完成。不需要懂 git checkout，不需要知道 HEAD 是什么意思。

第三件，让「敢用 AI」这件事有底气。这是最重要的一点。很多人用 AI 做事，到了关键文件、重要项目的时候会犹豫，怕 AI 改出问题。rew 解决的是这个心理层面的障碍，不是让你更谨慎，是让你更放得开。

这也是 rew 和现有工具最本质的差异，它在解决「如何让 AI 操作变得可见、可追溯、可撤销」，而不只是「出了事有个备份能找回来」。

回到具体功能，rew 的保护分三层。

**看着（Watch）**

rew 挂进 Claude Code、Cursor、WorkBuddy 等主流 AI 工具的 Hook 系统。每次你提交一个 Prompt，rew 就打开一个新的任务边界；每次工具调用结束，rew 就记录一次文件变更。整个过程在后台静默运行，你感觉不到它在工作。

不光是变更记录。rew 还会在后台维护一个文件基线，每个受保护文件在任务开始前的状态，都有一份快照存着。

**记着（Record）**

这是 rew 和所有同类工具最本质的差别。

打开任务详情，你能看到你当时输入的原始 Prompt 文本、AI 生成的这次变更的摘要、本次任务改了哪些文件（新建/修改/删除/重命名分类列出）、每个文件的行级 Diff，以及这次用的是哪个模型、调用了几次工具、任务耗时多长。

一周以后你看着任务列表，能清楚地知道「4月18日22:31那次，WorkBuddy帮我改了什么，当时让它做的是什么」。

**兜底（Restore）**

出了事，打开 rew，左侧是按时间倒序排列的 AI 任务列表，每条显示时间、AI 工具名称、Prompt 摘要、变更文件数。找到那条任务，右侧展开完整的文件 Diff，确认是这条没错，点「回到这一刻」。

rew 会对这次任务的每一条变更执行逆操作，新建的文件删掉，修改的文件回退到修改前，删除的文件从对象存储里恢复出来，重命名的文件把文件名和内容一起还原。

还可以精细操作，只恢复某一个文件，不影响其他改动。

## 三、用起来是什么感觉

官网下载，双击安装，全程不超过两分钟。

装完之后 rew 常驻系统托盘，一个小图标，实时防护中。日常完全无感，CPU 占用低于 1%，内存不超过 50MB，它安静地待在那里，你感觉不到它。

安装向导会问你要保护哪些目录，默认勾上桌面、文稿、下载，这三个地方是大多数人放「真正重要的东西」的地方。你也可以加入具体的项目目录。

然后执行一条命令：

```bash
rew install
```

rew 会自动检测你装了哪些 AI 工具，把 Hook 注入进去。Claude Code、Cursor、WorkBuddy、CodeBuddy，检测到什么注入什么。从这一刻起，这些工具的每次操作 rew 都知道。

我自己前几天有个真实的例子。在用 WorkBuddy 迭代 rew 的官网，改一个按钮的过渡动画。我的描述是「首先不要有黑色的缝隙和边线，让接触处由玻璃厚度变薄的透光感觉来体现」。AI 对这个描述理解有偏差，不只改了动画，连整个页面的配色、间距、按钮背景都一起动了。整个页面乱掉了。

以前遇到这种情况，要么翻 git stash，要么对着 Diff 一行行手动撤，要么重新描述一遍让 AI 改回去，还不一定改对。

这次打开 rew，左边任务列表找到那条「22:28 WorkBuddy」，右侧展开是 index.html 的 Diff，+31 行 -116 行，一眼就确认是这条没错，点「回到这一刻」，等了几秒，页面回来了。

前后不超过三十秒。

## 四、底下是怎么跑的

这部分给想深入了解的同学，我来聊聊 rew 在设计上遇到的几个真实问题，以及我是怎么解决的。

---

坦率的讲，我一开始设计 rew 的时候走了一条弯路。最直觉的思路是，直接监听文件系统事件不就好了，macOS 有 FSEvents，性能很好，文件一变就能知道。

但跑起来之后发现一个根本性的问题：FSEvents 只知道「文件变了」，不知道「是谁改的、为什么改的、这次改属于哪次 AI 对话」。你看到一堆文件变更事件，不知道它们是哪次 Prompt 触发的，更不知道要不要一起恢复。

所以 rew 最终用了两套并行的机制。

一套是 Hook 引擎，挂进 AI 工具的事件系统。当 AI 工具支持 Hook 时，比如 Claude Code、Cursor、WorkBuddy，rew 可以在四个时机介入，用户提交 Prompt 时开启一个新任务边界、工具调用前备份当前文件状态、工具调用后记录变更、AI 完成响应时把整次任务打包归档。这一套的粒度最细，因为知道每个文件是在哪次 Prompt 的第几次工具调用里被改的，以及改之前的内容是什么。

另一套就是 FSEvents 引擎，作为兜底。当 AI 工具没有 Hook 支持，或者 AI 通过子进程间接操作文件绕过了 Hook，FSEvents 这层还在。

目前已适配的工具：

| AI工具 | Hook路径 | 支持事件 |
|--------|---------|---------|
| Claude Code | `~/.claude/settings.json` | PreToolUse / PostToolUse / UserPromptSubmit / Stop |
| Cursor | `~/.cursor/hooks.json` | preToolUse / postToolUse / afterFileEdit / beforeSubmitPrompt / stop |
| WorkBuddy | `~/.workbuddy/settings.json` | PreToolUse / PostToolUse / UserPromptSubmit / Stop |
| CodeBuddy | `~/.codebuddy/settings.json` | PreToolUse / PostToolUse / UserPromptSubmit / Stop |
| 其他 | 不注入 | FSEvents 兜底 |

---

然后遇到了第二个问题，也是我觉得蛮有意思的设计挑战。

Hook 是 AI 工具同步调用的，它把事件扔给 rew，然后等 rew 返回，才继续往下走。也就是说，rew 处理 Hook 的速度，直接影响 AI 工具的响应速度。但 rew 要做的事不轻，入库、计算 Diff、解析 baseline，这些操作有一定耗时，不能放在同步路径上阻塞 AI。

我想了一段时间，解法是 Hook Spool，一个磁盘上的有序文件队列。

Hook 命令收到事件之后，只做一件事：把事件序列化写入 `pending/` 目录，然后立刻返回，整个过程几毫秒。AI 工具完全感知不到延迟。后台 daemon 异步消费这个队列，处理完了文件移到 `done/`，失败了移到 `failed/` 同时写一个 sidecar 错误文件方便排查。

还有一个细节是，如果 daemon 崩溃重启，`processing/` 状态里的事件会自动 requeue 回 `pending/`，不会因为进程崩溃而丢失数据。

```
pending/ → processing/ → done/
                      ↘ failed/ + error.txt（崩溃自动 requeue 回 pending）
```

---

顺着上面的再聊聊，多工具并发是另一个真实遇到的问题。

你同时开着 Cursor 和 WorkBuddy，两个 AI 都在处理你的问题，碰巧都改了同一个文件，rew 怎么知道每个任务「改之前」的状态是什么？

解法是 Pre-tool Hash Store，按 session_key 隔离的哈希存储。每次 PreToolUse 触发时，按 session_key（每个 AI 工具的每个会话有唯一标识）存一份当前文件的内容哈希。PostToolUse 进来时，先查这个 session 自己的 pre-tool hash 作为 baseline。

两个工具、两个 session，查的是各自的那份记录，互不干扰。恢复时每个任务还原的都是「自己那次操作之前」的状态，不会相互覆盖。

---

还有一个让我调试了一段时间的问题，是 Rename 的还原。

AI 重命名文件时，实际的操作往往不是原子性的 rename 系统调用，而是两步——删掉旧文件，创建新文件。Hook 捕获到的就是这两个独立事件，一个 Deleted，一个 Created。

如果直接按这两条记录恢复，结果是删掉新文件、创建旧文件，逻辑上对，但文件名没还原。

rew 在 Stop 事件之后有一个 reconcile 阶段，会对一次任务里的所有 Deleted + Created 做内容相似度匹配。旧文件和新文件内容高度相似的话，就把这两条合并成一条 Renamed 变更。

这样恢复的时候，还原的是完整的重命名操作，包括文件名，包括内容。

---

最后一个想单独说的是存储这块。

很多人担心备份会占满磁盘，这个顾虑我完全理解，所以特别说一下 rew 的存储机制。

rew 的文件备份用的是 macOS APFS 文件系统的 clonefile 机制。你可以把它理解成这样，创建一个文件的副本，不是把数据复制一遍，而是创建一个「指针」，两个文件名都指向同一块物理数据，只有当其中一个被写入修改时，才真正产生两份数据。

实际效果是，全量扫描 100 个文件并备份，磁盘占用接近 0，几乎是瞬间完成的。

加上 rew 用了 SHA-256 内容寻址的对象存储，相同内容的文件只存一份。同一个 `package.json` 被 10 次任务改来改去，实际存储只有 10 个不同版本的内容，不会每次都完整复制一遍。

这套存储架构的性能实测结果，16 万个文件，全量恢复，1 分钟内完成。

---

顺便说一下异常检测。rew 不只是被动记录，还内置了一套主动告警机制。当短时间内大量文件被删除、敏感配置文件被修改、或者受保护的根目录本身被删除时，系统会立即弹出系统通知，告诉你发生了什么、涉及了多少个文件、在哪个目录。

你可以选择「查看详情」直接跳到 rew 界面，或者「先不管」让快照保留备用。

| 规则 | 触发条件 | 级别 |
|------|---------|------|
| 批量删除 | 30秒内删除超过 20 个文件 | HIGH |
| 严重批量删除 | 1分钟内删除超过 50 个文件 | CRITICAL |
| 大量修改 | 短时间内大量文件被修改 | MEDIUM |
| 根目录删除 | 受保护根目录本身被删除 | CRITICAL |
| 敏感文件修改 | .env 等配置文件被修改 | HIGH |

## 五、和同类工具的对比

市面上有一个方向相近的工具叫 snaprevert，开源的 AI 文件监听工具，也是做「自动快照 + 一键回滚」，已有一定的社区用户。

我觉得它是个好工具，方向是对的，但两者的定位有本质差别。

snaprevert 用的是 Chokidar 文件监听 + diff 存储，没有 Hook 注入，也就不知道 Prompt 边界，只能靠时间 debounce（默认 3 秒）来猜测「这批变更可能属于同一次操作」。如果你的 Prompt 执行超过了 3 秒，它可能会把一次操作切成两条记录。它也没有图形界面，只有 CLI，也不处理 Rename 的语义还原。

| 维度 | snaprevert | rew |
|------|-----------|-----|
| 任务粒度 | 时间窗口 debounce | 精确 Prompt 边界 |
| Prompt 关联 | ✗ | ✓ 完整文本 |
| 多工具并发 | ✗ 无隔离 | ✓ session 隔离 |
| Rename 还原 | ✗ | ✓ reconcile 配对 |
| 界面 | CLI only | 桌面 GUI + CLI |
| 磁盘占用 | <10MB/天 | 接近 0（CoW） |

怎么选，如果你只需要一个轻量的命令行快照工具，snaprevert 够用。如果你用多个 AI 工具、想要清晰的操作历史、不想记命令，rew 更适合你。

## 七、能力边界：说清楚 rew 不能做什么

如果一篇文章只讲价值不讲边界，通常说明作者还没有真正把系统跑到真实现场。

rew 目前更适合这些场景，AI 工具在本机操作你的文件，改代码、改文档、整理目录，操作完了你想知道改了什么、或者想回退。这是 rew 的主战场。

而有几件事，现阶段就不应该过度承诺。

**只支持 macOS，需要 APFS 文件系统。** CoW 机制是 rew 性能的核心，APFS 是前提。2017 年之后的 Mac 默认就是 APFS，Windows 和 Linux 暂时不支持，这是明确的边界，不绕。

**不保护数据库。** rew 保护的是文件系统层的文件，SQLite、MySQL 这类数据库不在范围内。AI 删了你的数据库记录，rew 救不了。

**Hook 覆盖不到所有场景。** 如果 AI 通过 shell 执行系统命令间接操作文件（比如 `rm -rf`），这些操作走 FSEvents 兜底，能恢复，但没有 Prompt 上下文，这次操作会被记成「文件监听」任务。能回去，但看不到是哪次对话触发的。

**历史默认保留 30 天。** 超过后自动清理，可以通过 `~/.rew/config.toml` 调整。

坦率的讲，rew 现在是一个「把本机文件操作变得可见可回退」的工具，不是一个全面覆盖 AI 风险的完整解决方案。但我觉得这个范围本身已经很有价值，而且是真实可落地的范围，不是画饼。

## 八、最后

我做这个工具，有一部分是因为我自己出过事。

用 AI 做东西，状态好的时候会进入一种很流畅的节奏，一个 Prompt 接一个 Prompt，快速迭代。这种状态里，你不会想着停下来 commit，你只想继续往前走。

然后有一天 AI 干了一件它「理解正确」但你「完全不是这个意思」的事，你想回去，没有记录，没有存档。

游戏里的人敢冲 Boss，不是因为他不怕死。是因为他知道有存档，死了还能回来。存档这件事，让他更放得开，不是更谨慎。

rew 就是想做这个东西。不是让你对 AI 更小心，是让你更敢用。

目前只支持 macOS。官网下载，直接用，欢迎反馈。
