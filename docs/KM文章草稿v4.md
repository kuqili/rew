# 我给 AI 装了一个存档键

> **导语**
>
> 2025 年，AI 误删文件的事故在全球开发者社区里开始密集出现——不是因为 AI 变坏了，而是因为它变强了，能操控的范围越来越大，破坏半径也越来越大。出了事，评论区清一色是「你应该用 Git」，但这个答案对用 AI 整理文档的产品经理没用，对用 Cursor 做产品的设计师没用，对所有不是程序员的 AI 用户更没用。我自己也踩过两次，最惨那次找文件找了两个小时。
>
> 所以我做了 rew——一个挂进 AI 工具 Hook 系统、以每次用户 Prompt 为粒度自动存档、出了事一键回到任意时间点的工具。**16 万个文件，全量恢复，1 分钟内完成。** 本文会聊清楚它是什么、为什么这么设计、以及做的过程里踩了哪些真实的坑。

---

## 一、从一个很小的事故开始

故事是这样的。

2025 年 7 月，一个产品经理在用 Google Gemini CLI 做一件很小的事，把一个文件夹改个名字。

AI 开始执行，创建新目录，然后把旧目录里的文件一个一个移过去。

但有一步失败了。

创建新目录的命令，实际上没有成功执行，但 AI 的内部状态把它记成了「已成功」。它持续往一个从未存在过的目录里发移动指令。Windows 系统的行为是，当你把文件移动到一个不存在的路径时，它不报错，而是把这个文件重命名成那个路径名本身。每一条后续的移动命令，都把前一个文件覆盖掉。

最后，文件夹里只剩一个文件。

其他的，物理消失了。

AI 事后写了一行自我评价，原话是「I have failed you completely and catastrophically.」

## 二、然后是一个大得多的事故

在公司内部我也看到了类似的案例。

清明假期第一天，有个非技术背景的同学在家远程连公司电脑，用 CodeBuddy 开发一个线下数据计算工具。开发完了，想整理一下工程目录，就对 AI 说了一句：「请清空所有测试数据，让工具处于初始化状态。」

一句话。非常正常的一句话。

然后 AI 自行推导路径，生成清理脚本，不仅清空了工具的测试数据，还把他桌面和文档目录下的文件一并删除了。而他的桌面，恰好映射在 D 盘。那里存着他 2012 年入职至今，十三年，每一年的全部工作文件。

事后用专业工具扫描，文件名和目录结构还在，但固态硬盘 TRIM 已开启，恢复出来的文件没有一个能打开。

十三年，归零。

## 三、这不是个例，这是系统性问题

我当时看到这些，第一反应是，这是个例吧。某个工具的 bug，某个用户运气差，一次性事件。

然后我去翻了一下，越翻越心惊。

2026 年 3 月，Claude Code 被人发现，会每隔大约 10 分钟，静默执行一次 `git reset --hard origin/main`。你所有没 commit 的代码，每 10 分钟可能被悄悄清空一次。你完全不知道。这个 bug 从 2025 年 9 月就有人报了，issue 被关了，没修。

同样是 3 月，Cursor 出了一个代码静默回退的 bug，原因是 Agent Review 和云同步之间的竞态条件。有用户说自己丢了 4 个月的工作。

4 个月。。。

还有人发帖说，他在 CLAUDE.md 里明确写了「禁止执行任何破坏性 git 命令」。Claude 看完了这条规则，20 分钟后，在同一个 session 里，执行了被禁止的命令，覆盖了 30 多个文件的手动编辑。

你亲手写的规则，它看了，记了，然后转头就违反了。

你敢信？？？

这不是偶发事故。这是一个正在蔓延的系统性问题。

## 四、「用 Git 就好了」，这个答案错在哪里

那篇 13 年文件归零的帖子，我当时看了一遍评论区。

清一色都是「你应该用 Git」。

我完全理解这种直觉反应。Git 是开发者解决「文件搞乱了怎么办」的标准答案，用了这么多年，肌肉记忆一样。

但这个答案，在这件事上是错的。

首先，这个用户在用 AI 帮他清理文档，不是在写代码。你不可能把 `~/Documents` 整个 init 成一个 Git 仓库去管理，性能上不行，概念上也混乱。

其次，评论区那些人大概率都是技术背景，他们忘了一件事——现在用 AI 工具的，绝大多数不是程序员。63% 的 vibe coding 用户不是程序员，这个群体的同比增速是 520%，Cursor 的估值飙到 92 亿美金，整个市场已经 47 亿美金了。

让产品经理、运营同学、设计师去学 Git，去理解 commit、branch、checkout，这个学习成本不低，而且完全偏离了他们用 AI 的初衷。

还有一点，就算你是技术背景，你在用 AI 编程时，也会有这样的时刻：一个 Prompt 接一个 Prompt，状态飞速，根本不想打断自己去 commit。AI 一分钟能改几十个文件，等你发现问题的时候，那段空白期在 git log 里什么都没有。

而且，Claude Code 自己会偷偷执行 `git reset --hard`。你的安全网本身，被 AI 给破坏了。

---

## 五、真正的问题是一个速度差

我觉得根本原因在一个更底层的地方。

传统的版本管理工具，Git 也好，Time Machine 也好，它们的设计假设是「人类以人类的速度修改文件」。你花 30 分钟改两个文件，停下来想想，commit 一下，喝口水，继续。这个节奏下，手动 commit 完全够用。

但 AI 不是这样的。

AI 是你说一句话，30 秒内改了 15 个文件。你还没看清改了什么，下一轮 Prompt 又来了，又改了 10 个文件。5 轮下来，你想回到第 3 轮的状态，但那个状态，从来没有被任何地方记录过。

这是一个**速度差**的问题。AI 以机器的速度操作，人类以人类的速度管理，中间的间隙里，什么保护都没有。

现有工具的缺口，用一张表看清楚：

| 方案 | AI操作粒度 | 批量文件 | 零学习成本 | 知道是AI干的 |
|------|-----------|---------|-----------|------------|
| Git | ✗ 手动commit | ✓ | ✗ 需学命令 | ✗ |
| Time Machine | ✗ 小时级 | ✓ | ✓ | ✗ |
| VSCode History | ✗ 保存级 | ✗ 单文件 | ✓ | ✗ |
| **rew** | **✓ 每次Prompt** | **✓** | **✓** | **✓** |

最关键的缺口是最后一列。没有任何方案知道「这是 AI 干的」，也就没法按 AI 操作的粒度来回溯。

## 六、所以我决定做 rew

我自己也踩过两次。

去年在用 Claude Code 做项目，让它「把所有旧的 API 调用方式统一替换成新的」。它做了，很彻底也很快。然后中间提交了三次 commit，发现某个功能坏掉了，追回去是那次批量替换搞的，但三个 commit 里还有我想保留的改动。cherry-pick 来 cherry-pick 去，花了将近一个小时。

还有一次，AI 帮我「整理了一下项目结构」，删了一些它认为「没有被引用」的文件，但里面有几个确实有用，只是没被静态分析到。那几个文件没 commit 过，在 git 里压根不存在，只能靠 IDE 本地历史一个个找，找了两个小时。

就是那次之后，我决定做 rew。

## 七、rew 是什么

rew，取自 rewind，倒带。

**AI 时代下，你做的每一件事，都可以回到过去任意一刻。**

一句话描述它的定位，它是 AI 操作的可视化时间线，加上任意时间点的一键恢复。

不是版本管理，不是备份软件，不是沙箱拦截。它不会阻止 AI 做任何事，只做一件事——在 AI 动你文件的时候悄悄记下来，出了事找到那一刻，回去。

我想了很久才找到这四个字，**善意破坏**。AI 不是恶意的，它真的在努力帮你做事，它只是理解错了你的意思，然后以机器的速度执行了错误的操作。你让实习生整理文件夹，就算他理解错了，一分钟最多动几个文件，你看到了可以叫停。但 AI 不一样，3 秒内可能删了 800 个文件，你反应过来的时候，SSD 已经 TRIM 了。

这是一个只有 AI 时代才存在的问题，所以需要一个专门为它设计的解决方案。

## 八、三层保护，分别解决什么

先给结论：rew 真正在解决的是三件事，不只是「出了事有个备份」。

**看着（Watch）**，AI 每次动你的文件，自动存档，按 Prompt 粒度分组。rew 挂进 Claude Code、Cursor、WorkBuddy 等主流 AI 工具的 Hook 系统。每次你提交一个 Prompt，rew 就打开一个新的任务边界；每次工具调用结束，记录一次文件变更。整个过程后台静默，你感觉不到它在工作。

**记着（Record）**，这是 rew 和所有同类工具最本质的差别。打开任务详情，你能看到原始 Prompt 文本、AI 生成的变更摘要、改了哪些文件（新建/修改/删除/重命名分类列出）、每个文件的行级 Diff，以及这次用的是哪个模型、调用了几次工具、任务耗时多长。不只是「文件变了」，是「你让 AI 做什么、AI 做了什么」，这是 Git 和 Time Machine 给不了的。

**兜底（Restore）**，出了事，左侧任务列表找到那条，右侧展开完整 Diff，确认没错，点「回到这一刻」，等几秒，完成。Created 的文件删掉，Modified 的回退，Deleted 的从对象存储恢复，Renamed 的文件名和内容一起还原。不需要懂 git checkout，不需要知道 HEAD 是什么意思。

## 九、用起来是什么感觉

官网下载，双击安装，两分钟搞定。

装完常驻系统托盘，实时防护中。CPU 占用低于 1%，内存不超过 50MB，日常完全无感。

执行一条命令：

```bash
rew install
```

rew 自动检测你装了哪些 AI 工具，把 Hook 注入进去。Claude Code、Cursor、WorkBuddy、CodeBuddy，检测到什么注入什么，从这一刻起这些工具的每次操作 rew 都知道。

前几天我在用 WorkBuddy 改官网的按钮动画，描述是「不要有黑色的缝隙，让接触处由玻璃厚度变薄的透光感觉来体现」。AI 理解偏了，不只改了动画，连配色、间距、按钮背景都一起动了，整个页面乱掉了。

打开 rew，左边找到那条「22:28 WorkBuddy」，右侧是 index.html 的 Diff，+31 行 -116 行，确认没错，点「回到这一刻」，几秒钟，页面回来了。前后不超过三十秒。

## 十、底下是怎么跑的

这部分给想深入了解的同学看，我来聊聊设计上遇到的几个真实问题。

坦率的讲，我一开始走了一条弯路。最直觉的思路是直接监听文件系统事件，macOS 有 FSEvents，性能好，文件一变就能知道。但跑起来之后发现，FSEvents 只知道「文件变了」，不知道「是谁改的、为什么改的、属于哪次 AI 对话」。你看到一堆变更事件，不知道它们是哪次 Prompt 触发的。

所以 rew 最终用了两套并行机制。一套是 Hook 引擎，挂进 AI 工具的事件系统，在 Prompt 提交 / 工具调用前 / 工具调用后 / 任务结束四个时机介入，粒度最细。另一套是 FSEvents 引擎，当 AI 工具没有 Hook 支持，或通过子进程绕过 Hook，FSEvents 兜底。

| AI工具 | Hook路径 | 支持事件 |
|--------|---------|---------|
| Claude Code | `~/.claude/settings.json` | PreToolUse / PostToolUse / UserPromptSubmit / Stop |
| Cursor | `~/.cursor/hooks.json` | preToolUse / postToolUse / afterFileEdit / beforeSubmitPrompt / stop |
| WorkBuddy | `~/.workbuddy/settings.json` | PreToolUse / PostToolUse / UserPromptSubmit / Stop |
| CodeBuddy | `~/.codebuddy/settings.json` | PreToolUse / PostToolUse / UserPromptSubmit / Stop |
| 其他 | 不注入 | FSEvents 兜底 |

然后遇到了第二个问题，Hook 是 AI 工具同步调用的，等 rew 返回才继续走，rew 不能慢，但入库/Diff 计算/baseline 解析都有耗时。解法是 Hook Spool，一个磁盘上的有序文件队列。Hook 收到事件只做一件事：写入 `pending/` 目录，立刻返回，几毫秒。后台 daemon 异步消费，处理完移到 `done/`，失败移到 `failed/`。daemon 崩溃重启，`processing/` 里的事件自动 requeue，不丢数据。

```
pending/ → processing/ → done/
                      ↘ failed/ + error.txt（崩溃自动 requeue 回 pending）
```

多工具并发改同一个文件，baseline 怎么算？用 Pre-tool Hash Store，按 session_key 隔离，PreToolUse 时各存一份文件哈希，PostToolUse 时各查自己的那份，互不干扰。

Rename 怎么还原？AI 重命名往往是删旧文件 + 创建新文件两步。rew 在 Stop 之后有一个 reconcile 阶段，对 Deleted + Created 做哈希相似度匹配，配对成功合并成一条 Renamed 变更，恢复时文件名和内容一起还原。

## 十一、存储为什么不会撑爆磁盘

很多人担心这个，所以单独说。

rew 的文件备份用的是 macOS APFS 的 clonefile 机制，写时复制（CoW）。创建副本时不复制数据，只复制元数据，两个文件名指向同一块物理数据，写入时才真正分叉。

实际效果是全量扫描 100 个文件并备份，磁盘占用接近 0，几乎瞬间完成。

加上 SHA-256 内容寻址对象存储，相同内容只存一份。同一个 `package.json` 被 10 次任务改来改去，实际存储只有 10 个不同版本，不会每次完整复制。

性能实测：**16 万个文件，全量恢复，1 分钟内完成。**

## 十三、能力边界，不画饼

如果一篇文章只讲价值不讲边界，通常说明作者还没有真正把系统跑到真实现场。

rew 目前的主战场是：AI 工具在本机操作你的文件，改代码、改文档、整理目录，操作完了你想知道改了什么、或者想回退。

几件事，现阶段不应该过度承诺。

**只支持 macOS，需要 APFS 文件系统。** 2017 年之后的 Mac 默认是 APFS，Windows 和 Linux 暂时不支持。

**不保护数据库。** SQLite、MySQL 这类数据库不在范围内，AI 删了数据库记录，rew 救不了。

**Hook 不能覆盖所有场景。** AI 通过 shell 间接操作文件（比如 `rm -rf`），FSEvents 能捕获文件变更，但拿不到 Prompt 上下文，能恢复，但看不到是哪次对话触发的。

**历史默认保留 30 天，** 可通过 `~/.rew/config.toml` 调整。

坦率的讲，rew 现在是一个「把本机文件操作变得可见可回退」的工具，不是全面覆盖 AI 风险的完整解决方案。但这个范围本身已经很有价值，是真实可落地的范围，不是画饼。

## 十四、最后

1959 年，沃尔沃发明了三点式安全带，然后做了一个在商业史上很牛的决定——把这个专利开放了，免费给所有汽车厂商用。

从此安全带成了每辆车的标配。

不是因为安全带让车跑得更快，是因为安全带让人敢开更快。

2026 年的 AI 行业，所有人都在卷一件事，让 AI 更强、更快、更聪明。这当然重要。但很少有人在认真想另一个问题，怎么让人，敢用 AI。

信任比能力更稀缺。

rew 就是想做这个东西。不是让你对 AI 更小心，是让你更敢用。
