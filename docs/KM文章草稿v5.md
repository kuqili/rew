# 我给 AI 装了一个存档键

> **导语**
>
> 2025 年，AI 误删文件的事故在全球开发者社区里开始密集出现——不是因为 AI 变坏了，而是因为它变强了，能操控的范围越来越大，破坏半径也越来越大。出了事，评论区清一色是「你应该用 Git」，但这个答案对用 AI 整理文档的产品经理没用，对用 Cursor 做产品的设计师没用，对所有不是程序员的 AI 用户更没用。我自己也踩过两次，最惨那次找文件找了两个小时。
>
> 所以我做了 rew——一个挂进 AI 工具 Hook 系统、以每次用户 Prompt 为粒度自动存档、出了事一键回到任意时间点的工具。**16 万个文件，全量恢复，1 分钟内完成。** 本文会聊清楚它是什么、为什么这么设计、以及做的过程里踩了哪些真实的坑。

---

## 一、这不是事故，这是 AI 时代的新常态

### 1.1 文件夹改名，其他的物理消失了

故事是这样的。

2025 年 7 月，一个产品经理在用 Google Gemini CLI 做一件很小的事，把一个文件夹改个名字。

任务本身没什么难度。AI 开始执行，创建新目录，然后把旧目录里的文件一个一个移过去。

但有一步失败了。

创建新目录的命令，实际上没有成功执行，但 AI 的内部状态把它记成了「已成功」。从这一刻起，AI 的「世界模型」和真实的文件系统彻底脱轨了。它持续往一个从未存在过的目录里发移动指令。

Windows 系统的行为是，当你把文件移动到一个不存在的路径时，它不报错，而是把这个文件重命名成那个路径名本身。每一条后续的移动命令，都把前一个文件覆盖掉。

最后，文件夹里只剩一个文件。

其他的，物理消失了。

AI 事后写了一行自我评价，原话是「I have failed you completely and catastrophically.」

> 出处：Ars Technica [《Two major AI coding tools wiped out user data after making cascading mistakes》](https://arstechnica.com/information-technology/2025/07/ai-coding-assistants-chase-phantoms-destroy-real-user-data/)（2025-07-24）；Mashable [《Google Gemini deletes user's code》](https://me.mashable.com/tech/58798/google-gemini-deletes-users-code-i-have-failed-you-completely-and-catastrophically)（2025-07-25）

### 1.2 十三年，归零

。。。

在公司内部我也看到了类似的案例。

清明假期第一天，有个非技术背景的同学在家远程连公司电脑，用 CodeBuddy 开发一个线下数据计算工具。开发完了，想整理一下工程目录，就对 AI 说了一句：

「请清空所有测试数据，让工具处于初始化状态。」

一句话。非常正常的一句话。

然后 AI 自行推导路径，生成清理脚本，不仅清空了工具的测试数据，还把他桌面和文档目录下的文件一并删除了。而他的桌面，恰好映射在 D 盘。那里存着他 2012 年入职至今，十三年，每一年的全部工作文件。

事后用专业工具扫描，文件名和目录结构还在，但固态硬盘 TRIM 已开启，恢复出来的文件没有一个能打开。

十三年，归零。

### 1.3 然后我发现，这根本不是个例

我当时看到这些，第一反应是，这是个例吧。某个工具的 bug，某个用户运气差，一次性事件。

然后我去翻了一下，越翻越心惊。

2025 年 8 月，有人在 Claude Code 的 GitHub 仓库提了一个标记为 CRITICAL 的 bug（issue #5516）。他在 `CLAUDE.md` 里明确写了：

```
NEVER EDIT /whaticket/backend/*
NEVER EDIT /whaticket/frontend/*
```

Claude 读了这条规则，确认了，然后在同一个 session 里，修改了被明令禁止的目录，破坏了生产环境里正在运行的 API 集成。用户警告它，它道歉，承诺不再犯，然后下一轮继续犯。Anthropic 最终把这个 issue 标记为「非计划内」，关掉了。

你亲手写的规则，它看了，记了，然后转头就违反了。

你敢信？？？

这不是孤例。同类 issue 还有 #5055、#2544、#2901，记录着同样的模式。

> 出处：[anthropics/claude-code #5516](https://github.com/anthropics/claude-code/issues/5516)（2025-08-10，已关闭，标记为「非计划内」）

2026 年 3 月，Cursor 官方论坛确认了一个代码静默回退的 bug。原因是 Agent Review Tab 和云同步之间的竞态条件——云端的旧版本会悄悄覆盖你本地最新的改动，没有任何提示，没有任何警告。有开发者记录了自己 4 个月的工作因此丢失。官方给出的「临时解决方案」是：使用某个功能之前，先手动关掉另一个功能。

> 出处：[Cursor Problems (2026)](https://vibecoding.app/blog/cursor-problems-2026)（Cursor 官方论坛确认，Medium @tahabebek 记录了 4 个月工作丢失案例）

这不是偶发事故。这是一个正在蔓延的系统性问题。

## 二、为什么现有工具都不够用

### 2.1 「用 Git 就好了」，这个答案错在哪里

那篇 13 年文件归零的帖子，我当时看了一遍评论区。

清一色都是「你应该用 Git」。

我完全理解这种直觉反应。Git 是开发者解决「文件搞乱了怎么办」的标准答案，用了这么多年，肌肉记忆一样。

但这个答案，在这件事上是错的。

首先，那个用户在用 AI 帮他清理文档，不是在写代码。你不可能把 `~/Documents` 整个 init 成一个 Git 仓库去管理，性能上不行，概念上也混乱。Git 是为代码仓库设计的，不是为「用户的整台电脑」设计的。

其次，评论区那些人大概率都是技术背景，他们忘了一件事——现在用 AI 工具的，绝大多数不是程序员。63% 的 vibe coding 用户不是程序员，这个群体的同比增速是 520%，Cursor 的估值飙到 92 亿美金，整个 vibe coding 市场已经 47 亿美金了。这个问题正在随着用户规模的扩大快速放大

让产品经理、运营同学、设计师去学 Git，去理解 commit、branch、checkout，这个学习成本不低，而且完全偏离了他们用 AI 的初衷。

最后还有一点，Claude Code 自己会偷偷执行 `git reset --hard`。你的安全网本身，被 AI 给破坏了。

### 2.2 真正的问题是一个速度差

我觉得根本原因在一个更底层的地方。

传统的版本管理工具，Git 也好，Time Machine 也好，它们的设计假设是「人类以人类的速度修改文件」。你花 30 分钟改两个文件，停下来想想，commit 一下，喝口水，继续。这个节奏下，手动 commit 完全够用。

但 AI 不是这样的。

AI 是你说一句话，30 秒内改了 15 个文件。你还没看清改了什么，下一轮 Prompt 又来了，又改了 10 个文件。5 轮下来，你想回到第 3 轮的状态，但那个状态，从来没有被任何地方记录过。

这是一个**速度差**的问题。AI 以机器的速度操作，人类以人类的速度管理，中间的间隙里，什么保护都没有。

### 2.3 现有方案缺口一览

用一张表看清楚：

| 方案 | AI操作粒度 | 批量文件 | 零学习成本 | 知道是AI干的 |
|------|-----------|---------|-----------|------------|
| Git | ✗ 手动commit | ✓ | ✗ 需学命令 | ✗ |
| Time Machine | ✗ 小时级 | ✓ | ✓ | ✗ |
| VSCode History | ✗ 保存级 | ✗ 单文件 | ✓ | ✗ |
| **rew** | **✓ 每次Prompt** | **✓** | **✓** | **✓** |

最关键的缺口是最后一列。没有任何方案知道「这是 AI 干的」，也就没法按 AI 操作的粒度来回溯。这不是某个工具做得不够好，是所有这些工具都不是为这件事设计的。

## 三、所以我做了 rew

### 3.1 我自己踩过的两次

去年在用 Claude Code 做项目，状态特别好，一个 Prompt 接一个 Prompt，飞速往前。有一次让它「把项目里所有旧的 API 调用方式统一替换成新的」。

它做了。很彻底，也很快。

然后中间加了很多新的改动，提交了三次 commit。发现某个功能完全坏掉了，追了半天，发现是那次批量替换把某个边界情况也改掉了，但那个边界情况的逻辑其实是对的。

我想回退那次操作。

git log 一看，那次替换发生在三个 commit 之前，但那三个 commit 里还有我想保留的改动。cherry-pick 来 cherry-pick 去，最终花了将近一个小时，把该回的退了，把不该动的保住了。

我当时就想，如果我能直接看到「这次 AI 操作改了什么」，然后「只撤销这一次」，该多好。

后来又出了一次，AI 帮我「整理了一下项目结构」，删了一些它认为「没有被引用」的文件，但里面有几个确实有用，只是当时没有被静态分析到。那几个文件没有 commit 过，在 git 里压根不存在，只能靠 IDE 本地历史一个个找回来，找了两个小时。

就是那次之后，我决定做 rew。

### 3.2 rew 是什么

rew，取自 rewind，倒带。

**AI 时代下，你做的每一件事，都可以回到过去任意一刻。**

一句话描述它的定位，它是 AI 操作的可视化时间线，加上任意时间点的一键恢复。不是版本管理，不是备份软件，不是沙箱拦截。它不会阻止 AI 做任何事，只做一件事，在 AI 动你文件的时候悄悄记下来，出了事找到那一刻，回去。

#### 3.2.1 善意破坏

我想了很久才找到这四个字，**善意破坏**。

AI 不是恶意的。它真的在努力帮你做事，它只是理解错了你的意思，然后以机器的速度执行了错误的操作。

你让实习生帮你整理文件夹，就算他理解错了，一分钟最多动几个文件，你看到了可以叫停。但 AI 不一样，你让它帮你清数据，3 秒内可能删了 800 个文件，你反应过来的时候，SSD 已经 TRIM 了。

这是一个只有 AI 时代才存在的问题，所以需要一个专门为它设计的解决方案。

### 3.3 先给结论：rew 解决的三件事

如果只看界面截图，rew 很容易被理解成一个「自动备份工具」。但它真正在解决的，是三个具体的、高频的场景。

**场景一，AI 误删了你的文件，怎么找回来。**

这是最极端的情况，也是最容易被忽视的。AI 执行删除操作不会进回收站，`rm` 命令删的文件就是没了，SSD TRIM 开启之后专业工具也扫不出来。rew 在 AI 动文件之前就把内容存了一份，出了事从对象存储里还原，不依赖回收站，不依赖文件系统的删除记录。

**场景二，跟 AI 来回改了好多轮，想回到中间某一个版本。**

这是日常最高频的场景。你和 AI 一起迭代一个功能，第一轮还不错，第二轮有点偏，第三轮更偏，第四轮干脆乱了，你想回到第一轮结束的那个状态。

但第一轮没有 commit，第二轮也没有，你只想继续往前跑，不想打断自己去提交。现在想回去，git log 里什么都没有。

rew 以每次 Prompt 为粒度自动存档，左边时间轴找到那条，右边看 Diff，点「回到这一刻」，完成。不需要你在状态好的时候打断自己去提交，rew 替你记着。

**场景三，不想再主动管版本，让 AI 自己跑就好。**

AI 改代码的速度远远快过人提交代码的速度。如果你要求自己每次 AI 改完都去 commit，那你要么跟不上，要么打断自己的心流，要么就懒得提交了——然后哪天出了事什么都没有。

rew 的存在让你可以彻底不管版本提交这件事，它在后台盯着，你只管往前冲。Git commit 继续做，但不再是你的安全网，它是你的历史记录；rew 才是你在 commit 之间的那道保险。

## 四、三层保护的具体实现

### 4.1 看着（Watch）

rew 挂进 Claude Code、Cursor、WorkBuddy 等主流 AI 工具的 Hook 系统。每次你提交一个 Prompt，rew 就打开一个新的任务边界；每次工具调用结束，记录一次文件变更。整个过程后台静默运行，你感觉不到它在工作。

不光是变更记录，rew 还会在后台维护一个文件基线，每个受保护文件在任务开始前的状态，都有一份快照存着，这是后续恢复的基础。

### 4.2 记着（Record）

这是 rew 和所有同类工具最本质的差别。

打开任务详情，你能看到你当时输入的原始 Prompt 文本、AI 生成的这次变更摘要、本次任务改了哪些文件（新建/修改/删除/重命名分类列出）、每个文件的行级 Diff，以及这次用的是哪个模型、调用了几次工具、任务耗时多长。

一周以后你看着任务列表，能清楚地知道「4 月 18 日 22:31 那次，WorkBuddy 帮我改了什么，当时让它做的是什么」。这是 Git 和 Time Machine 给不了的，因为它们记录的是你主动提交的内容，而不是 AI 做了什么。

### 4.3 兜底（Restore）

出了事，打开 rew，左侧是按时间倒序排列的 AI 任务列表，每条显示时间、AI 工具名称、Prompt 摘要、变更文件数。找到那条任务，右侧展开完整的文件 Diff，确认是这条没错，点「回到这一刻」。

rew 会对这次任务的每一条变更执行逆操作，新建的文件删掉，修改的文件回退到修改前，删除的文件从对象存储里恢复出来，重命名的文件把文件名和内容一起还原。

还可以精细操作，只恢复某一个文件，不影响其他改动。

### 4.4 异常检测（Alert）

rew 不只是被动记录，还内置了一套主动告警机制。当短时间内大量文件被删除、敏感配置文件被修改、或者受保护的根目录本身被删除时，系统会立即弹出系统通知，告诉你发生了什么、涉及了多少个文件、在哪个目录。

你可以选择「查看详情」直接跳到 rew 界面，或者「先不管」让快照保留备用。

| 规则 | 触发条件 | 级别 |
|------|---------|------|
| 批量删除 | 30秒内删除超过 20 个文件 | HIGH |
| 严重批量删除 | 1分钟内删除超过 50 个文件 | CRITICAL |
| 大量修改 | 短时间内大量文件被修改 | MEDIUM |
| 根目录删除 | 受保护根目录本身被删除 | CRITICAL |
| 敏感文件修改 | .env 等配置文件被修改 | HIGH |

## 五、用起来是什么感觉

### 5.1 安装：两分钟，然后消失在后台

官网下载，双击安装，全程不超过两分钟。

安装向导会问你要保护哪些目录，默认勾上桌面、文稿、下载，这三个地方是大多数人放「真正重要的东西」的地方。加入项目目录也可以，rew 会在后台对这些目录做初始扫描，建立文件基线。

然后执行一条命令：

```bash
rew install
```

rew 会自动检测你装了哪些 AI 工具，把 Hook 注入进去。Claude Code、Cursor、WorkBuddy、CodeBuddy，检测到什么注入什么。这一步不需要你做任何配置，也不需要了解 Hook 是什么。

从这一刻起，这些工具的每次操作 rew 都知道。

装完之后 rew 常驻系统托盘，一个小绿点，「rew 实时防护中」。日常完全无感，CPU 占用低于 1%，内存不超过 50MB，它安静地待在那里，你感觉不到它。

---

### 5.2 界面：你需要知道的三个区域

打开 rew 主界面，布局分三块。

**左侧边栏**，是导航和状态。顶部是防护状态指示，下面是四个视图入口，最近活动、AI 任务历史、使用洞察、全部目录。「最近活动」是默认视图，显示所有 AI 工具最近的操作流。「AI 任务历史」可以按工具筛选，只看 Cursor 的、只看 WorkBuddy 的。「使用洞察」里有每天用了多少次 AI、用了哪些工具、调用了多少次、花了多少 token，挺好玩的。

**中间任务列表**，是核心。每条记录代表一次 AI 操作，显示时间、工具名称、Prompt 摘要、改了多少文件、加了多少行删了多少行。这是你的 AI 操作时间轴。

**右侧详情面板**，点开某条任务之后展开。顶部是时间点和「回到这一刻」按钮，下面依次是 Prompt 原文、AI Summary、变更文件列表、行级 Diff。你在这里看清楚 AI 干了什么，然后决定要不要恢复。

---

### 5.3 场景一：AI 误删了文件，怎么找回来

这是最极端的情况，也是大多数人第一次想到 rew 的时机。

AI 删文件不走回收站。`rm` 命令删的、脚本批量清的、「整理目录」顺手删掉的，文件就是没了。SSD TRIM 开启之后，专业恢复工具也救不回来。

用 rew 的处理方式是这样的：

打开主界面，左边「最近活动」里找到出事前后的那条任务，点开，右边是完整的变更列表。找到被删的文件，点「恢复此文件」，等几秒。

文件回来了，内容是 AI 删它之前的状态。

rew 在 AI 动文件之前就用 APFS clonefile 存了一份内容，不依赖回收站，不依赖文件系统的删除日志，所以能找回来。

如果整个任务都需要撤销，点「回到这一刻」，这次任务里所有的变更全部逆操作，新建的文件删掉，修改的回退，删除的恢复，重命名的还原文件名。

---

### 5.4 场景二：来回改了好多轮，想回到某个历史版本

这个场景我自己最高频遇到。

和 AI 一起迭代一个功能，第一轮效果不错，然后继续改，改到第四轮发现越来越偏，想回到第一轮结束的状态。但第一轮没有 commit，第二轮也没有，git log 里一片空白。

rew 的做法：

左边任务列表是按时间倒序的，往下翻找到「第一轮」那条任务，右边看 Diff 确认是这条没错，点「回到这一刻」。

rew 会把从那条任务之后的所有变更全部撤销，回到那个时间点的文件状态。

如果只是某几个文件出了问题，不想撤销整个任务，也可以在变更列表里单独选文件恢复，精细到单个文件粒度。

---

### 5.5 场景三：不想再手动管版本，让后台盯着就好

这是最「懒」也最实用的用法。

以前用 AI 开发的时候，总有一种焦虑感——AI 改太快，我来不及 commit，万一出事了怎么办。所以要么频繁打断自己去提交，要么心里一直悬着。

装了 rew 之后这个焦虑消失了。

你只管和 AI 对话，一个 Prompt 接一个 Prompt，rew 在后台默默记着每一次。不需要打断，不需要提交，出了事打开 rew 就能找到那条记录。

git commit 还是正常做，该做的时候做，但不再是你的安全网了。它是你的版本历史；rew 是你在 commit 之间那段时间里的保险。

---

### 5.6 使用洞察：你和 AI 协作的全局视图

这是 rew 里一个容易被忽视但挺有价值的功能。

「使用洞察」面板里记录了你用各个 AI 工具的情况，按天/周/月统计，包括：

- 每个工具的使用频次和任务数
- 平均每次任务改了多少文件、多少行
- 调用了多少次工具
- 消耗了多少 token

这不只是数字，它让你对「AI 帮我做了多少事」有一个直观的感知。我看过自己某一天的数据，WorkBuddy 改了 47 个文件，累计 +2300 行 -1800 行。如果那天没有 rew，我完全不知道 AI 动了多少东西。

---

### 5.7 磁盘占用：为什么几乎不占空间

这是很多人在意的问题，先给结论：**正常使用下，rew 的磁盘占用极低，基本感知不到。**

原因在于 rew 的三层文件保护设计。

截图里那张表说清楚了核心逻辑：

| 层级 | 文件类型 | 代表示例 | 能否恢复 | 磁盘占用 |
|------|---------|---------|---------|---------|
| **Layer 0 完全排除** | 可再生产物 & 系统噪声 & 用户主动排除 | `node_modules/` `dist/` `.git/` `*.tmp` | ✗ | 0 |
| **Layer 1 全量保护** | 代码 / 配置 / 文档 / 小图片（< 50MB）| `.rs` `.ts` `.json` `.md` `.png` | ✓ 任意任务节点均可恢复 | 初始备份占用 ≈ 0（CoW 共享物理块）；后续修改每次新增一份差异量，极小 |
| **Layer 2 有限保护** | 大型音视频（≥ 50MB）| `.mp4` `.mov` `.wav` `.flac` | △ 仅最近 2 个存档节点可恢复 | 初始备份占用 ≈ 0（CoW）；后续修改每次新增一份，但自动清理旧版，最多保留 2 份 |

rew 不备份 `node_modules/`、`dist/`、`.git/`、`target/` 这些东西，这类目录文件多、变化快、随时可以重新生成，备份它们毫无意义。真正需要保护的，是你花时间写的代码、配置、文档、图片，这些才进 Layer 1。

Layer 1 的文件用 APFS clonefile 做备份，写时复制，初始状态两个文件名共享同一块物理数据，只有被修改时才真正占用新空间。所以初始扫描几万个文件，磁盘占用接近 0。

后续每次 AI 修改，rew 只存一份该文件的新版本内容，通过 SHA-256 内容寻址对象存储，相同内容只存一份。你的项目里有 100 个文件，AI 这次只改了 3 个，rew 只新增这 3 个文件的新版本，其他 97 个文件继续共享原来的物理块。

**自动清理策略**

rew 有分级的历史保留策略，默认配置如下：

| 阶段 | 保留策略 | 默认时长 |
|------|---------|---------|
| 最近 1 小时 | 所有任务节点全部保留 | 3600 秒 |
| 最近 24 小时 | 每小时保留一个节点 | 86400 秒 |
| 最近 30 天 | 每天保留一个节点 | 30 天 |
| 30 天以上 | 自动清理 | — |
| 异常检测触发的任务 | 保留时间 × 2 倍 | — |

这些参数都可以在 `~/.rew/config.toml` 里调整：

```toml
[retention_policy]
keep_all_secs = 3600        # 最近多少秒内全量保留（默认 1 小时）
keep_hourly_secs = 86400    # 最近多少秒内按小时保留（默认 24 小时）
keep_daily_secs = 2592000   # 最近多少秒内按天保留（默认 30 天）
anomaly_retention_multiplier = 2  # 异常任务保留时间倍数
```

另外，`max_file_size_bytes` 参数控制单个文件的备份上限，超过这个大小的文件直接跳过，不进入对象存储。默认不限制，如果你觉得大文件占用太多可以手动设置。

## 六、底下是怎么跑的

### 6.1 为什么不能只监听文件系统

坦率的讲，我一开始设计 rew 的时候走了一条弯路。最直觉的思路是，直接监听文件系统事件不就好了，macOS 有 FSEvents，性能好，文件一变就能知道。

但跑起来之后发现一个根本性的问题：FSEvents 只知道「文件变了」，不知道「是谁改的、为什么改的、这次改属于哪次 AI 对话」。你看到一堆文件变更事件，不知道它们是哪次 Prompt 触发的，更不知道要不要一起恢复。

所以 rew 最终用了两套并行的机制，一套是 Hook 引擎，挂进 AI 工具的事件系统，拿到精确的 Prompt 粒度边界；另一套是 FSEvents 引擎，当 AI 工具没有 Hook 支持，或通过子进程间接操作文件绕过 Hook 时，FSEvents 这层兜底。

目前已适配的工具：

| AI工具 | Hook路径 | 支持事件 |
|--------|---------|---------|
| Claude Code | `~/.claude/settings.json` | PreToolUse / PostToolUse / UserPromptSubmit / Stop |
| Cursor | `~/.cursor/hooks.json` | preToolUse / postToolUse / afterFileEdit / beforeSubmitPrompt / stop |
| WorkBuddy | `~/.workbuddy/settings.json` | PreToolUse / PostToolUse / UserPromptSubmit / Stop |
| CodeBuddy | `~/.codebuddy/settings.json` | PreToolUse / PostToolUse / UserPromptSubmit / Stop |
| 其他 | 不注入 | FSEvents 兜底 |

### 6.2 Hook 不能慢，所以用了 Spool

然后遇到了第二个问题，也是我觉得蛮有意思的设计挑战。

Hook 是 AI 工具同步调用的，它把事件扔给 rew，然后等 rew 返回，才继续往下走。也就是说，rew 处理 Hook 的速度，直接影响 AI 工具的响应速度。但 rew 要做的事不轻，入库、计算 Diff、解析 baseline，这些操作有一定耗时，不能放在同步路径上阻塞 AI。

解法是 Hook Spool，一个磁盘上的有序文件队列。

Hook 命令收到事件之后，只做一件事：把事件序列化写入 `pending/` 目录，然后立刻返回，整个过程几毫秒。AI 工具完全感知不到延迟。后台 daemon 异步消费这个队列，处理完了文件移到 `done/`，失败了移到 `failed/` 同时写一个 sidecar 错误文件方便排查。

```
pending/ → processing/ → done/
                      ↘ failed/ + error.txt
```

#### 6.2.1 容灾设计

这里有一个细节值得说。如果 daemon 崩溃重启，`processing/` 状态里的事件会自动 requeue 回 `pending/`，不会因为进程意外崩溃丢失数据。整个 Spool 的状态流转是单向的，不可回退，这保证了事件有且仅有一次被处理。

### 6.3 多工具并发改同一个文件

顺着上面的再聊聊，多工具并发是另一个真实遇到的问题。

你同时开着 Cursor 和 WorkBuddy，两个 AI 都在处理你的问题，碰巧都改了同一个文件，rew 怎么知道每个任务「改之前」的状态是什么？

解法是 Pre-tool Hash Store，按 `session_key` 隔离的哈希存储。每次 PreToolUse 触发时，按 `session_key`（每个 AI 工具的每个会话有唯一标识）存一份当前文件的内容哈希。PostToolUse 进来时，先查这个 session 自己的 pre-tool hash 作为 baseline。两个工具、两个 session，查的是各自的那份记录，互不干扰。

#### 6.3.1 为什么用文件系统而不是内存

Pre-tool Hash Store 存在磁盘上，不是内存里。原因是，如果进程崩溃，内存里的数据全丢，下次恢复时就找不到「改之前」的状态了。磁盘上的文件，进程崩溃重启后依然在，下次 PostToolUse 进来还能查到。

这是一个典型的「用磁盘换可靠性」的权衡，在这个场景下完全值得。

### 6.4 Rename 怎么还原

还有一个让我调试了一段时间的问题，是 Rename 的还原。

AI 重命名文件时，实际的操作往往不是原子性的 rename 系统调用，而是两步——删掉旧文件，创建新文件。Hook 捕获到的就是这两个独立事件，一个 Deleted，一个 Created。

如果直接按这两条记录恢复，结果是删掉新文件、创建旧文件，逻辑上对，但文件名没还原。

rew 在 Stop 事件之后有一个 reconcile 阶段，会对一次任务里的所有 Deleted + Created 做内容相似度匹配——如果旧文件和新文件的内容高度相似，就把这两条合并成一条 Renamed 变更。这样恢复的时候，还原的是完整的重命名操作，包括文件名，包括内容。

### 6.5 存储为什么不会撑爆磁盘

最后一个想单独说的是存储这块，很多人担心备份会占满磁盘，这个顾虑我完全理解。

rew 的文件备份用的是 macOS APFS 文件系统的 clonefile 机制，写时复制（CoW）。你可以把它理解成这样，创建一个文件的副本，不是把数据复制一遍，而是创建一个「指针」，两个文件名都指向同一块物理数据，只有当其中一个被写入修改时，才真正产生两份数据。

实际效果是，全量扫描 100 个文件并备份，磁盘占用接近 0，几乎是瞬间完成的。

加上 rew 用了 SHA-256 内容寻址的对象存储，相同内容的文件只存一份。同一个 `package.json` 被 10 次任务改来改去，实际存储只有 10 个不同版本的内容，不会每次都完整复制一遍。

这套存储架构的性能实测结果，**16 万个文件，全量恢复，1 分钟内完成。**

## 八、能力边界，不画饼

如果一篇文章只讲价值不讲边界，通常说明作者还没有真正把系统跑到真实现场。

### 8.1 只支持 macOS / APFS

CoW 机制是 rew 性能的核心，APFS 是前提。2017 年之后的 Mac 默认就是 APFS，Windows 和 Linux 暂时不支持，这是明确的边界，不绕。

### 8.2 不保护数据库

rew 保护的是文件系统层的文件，SQLite、MySQL 这类数据库不在范围内。AI 删了你的数据库记录，rew 救不了，这个需要数据库层面的备份方案来解决。

### 8.3 Hook 覆盖不到所有场景

如果 AI 通过 shell 执行系统命令间接操作文件（比如 `rm -rf`），这些操作走 FSEvents 兜底，能恢复文件内容，但拿不到 Prompt 上下文，这次操作会被记成「文件监听」任务。能回去，但看不到是哪次对话触发的。

### 8.4 历史默认保留 30 天

超过后自动清理最旧的任务，可以通过 `~/.rew/config.toml` 调整保留策略。

---

坦率的讲，rew 现在是一个「把本机文件操作变得可见可回退」的工具，不是一个全面覆盖 AI 风险的完整解决方案。但我觉得这个范围本身已经很有价值，而且是真实可落地的范围，不是画饼。
