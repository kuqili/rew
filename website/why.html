为什么需要 rew

AI 的善意
有时候会造成破坏

AI 不是恶意的。它只是理解错了你的意思，然后以机器的速度执行了错误的操作。

真实案例

十三年，归零

清明假期第一天，一位开发者在家远程连公司电脑，用 CodeBuddy 开发一个线下数据计算工具。开发完了，想整理一下工程目录，就对 AI 说了一句：

「请清空所有测试数据，让工具处于初始化状态」

一句话。非常正常的一句话。

然后 AI 自行推导路径，生成清理脚本，不仅清空了工具的测试数据，还把他桌面和文档目录下的文件一并删除了。

而他的桌面，恰好映射在 D 盘。那里存着他 2012 年入职至今，十三年，每一年的全部工作文件。

事后用专业工具扫描，文件名和目录结构还在，但固态硬盘 TRIM 已开启，恢复出来的文件没有一个能打开。

十三年，归零。

这不是个例。类似的事情，每天都在发生。

事故还原
AI 越界删除事件
14:23
开发者发出指令：「清空所有测试数据」
14:23 +2s
AI 自行推导路径，执行范围超出工程目录
14:23 +3s
桌面 + 文档目录共计 800+ 文件被静默删除
数分钟后
SSD TRIM 执行，物理层面不可恢复
结果：2012–2025 年，十三年工作文件全部永久丢失。

现有方案的缺口

你以为有保护，其实没有

Git

需要主动 commit，需要学分支、暂存区。对非技术用户来说，这不是解决方案。更可怕的是：AI 会静默执行 git reset --hard。

需要手动 commit / 可被 AI 操控
Time Machine

按小时备份。但 AI 交互是按秒的。一小时内 AI 可能改了 50 次文件，Time Machine 只有一个快照。

时间粒度太粗
Dropbox / 云盘

有延迟，只管同步文件夹内的文件。AI 在终端里 rm 掉的文件？管不了。

覆盖范围有限
回收站

rm 命令删除的文件不进回收站。覆盖写入也不进。AI agent 在终端里的操作一概管不了。

Terminal 操作完全绕过
问题的本质：速度差

传统版本管理的设计假设是「人类以人类的速度修改文件」。AI 打破了这个假设。

沃尔沃安全带的故事

最早的汽车没有安全带。后来事故多了。1959 年，沃尔沃发明了三点式安全带，然后把专利开放，免费给所有厂商用。

从此安全带成了每辆车的标配。不是因为它让车跑得更快，是因为它让人敢开更快。

AI 行业现在也到了需要安全带的时候了。不缺更强的模型，缺的是让人敢用的安全网。
